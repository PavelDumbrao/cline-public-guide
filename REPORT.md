# Как создавался этот репозиторий

> История для тех, кто хочет понять не только «что», но и «почему именно так».

---

## С чего всё началось

Cline — мощный инструмент. Но у большинства людей, которые его впервые запускают, есть одна общая проблема: они не знают, как его настраивать.

Официальная документация объясняет механику. Но она не объясняет **стратегию**: какие правила писать в первую очередь, какие MCP ставить, а какие лучше не трогать поначалу, почему система ведёт себя непредсказуемо и как это исправить.

В клубе ProAiCommunity мы прошли через всё это на практике. Через хаос, через эксперименты, через десятки итераций.

Этот репозиторий — дистилляция того, что реально работает.

---

## Что было сделано

Создан публичный обучающий шаблон с полной структурой настройки Cline:

- `README.md` — точка входа с маршрутами для разных типов пользователей
- `docs/` — человекочитаемые гайды: от базовой концепции до работы с MCP и n8n
- `ai/rules/` — шаблоны правил поведения агента
- `ai/skills/` — специализированные навыки для конкретных задач
- `ai/hooks/` — примеры хуков на события Cline
- `ai/workflows/` — markdown-процессы для сложных многошаговых задач
- `examples/` — готовые промпты, smoke-тесты, примеры n8n workflows

---

## Что было намеренно исключено

Это публичный репозиторий. Поэтому в нём нет:

- личного `USER.md` и `tech-stack.md` — они уникальны для каждого
- живых токенов, API-ключей, IP-адресов и доменов
- приватных заметок об инцидентах
- инфраструктурных скриптов с привязкой к конкретному серверу
- n8n workflow JSON с реальными credentials

Всё, что могло содержать личную информацию, было либо обезличено через плейсхолдеры, либо удалено.

---

## Что пришлось придумать заново

Когда начали готовить репозиторий к публикации, стало ясно: просто скопировать живую систему нельзя.
Она была заточена под конкретного человека с конкретной инфраструктурой.

Пришлось:
- вычленить универсальные принципы из личных правил
- переписать skills так, чтобы они работали без зависимости от конкретных сервисов
- разделить «обязательное» и «опциональное» — чего раньше явно не было
- добавить маршруты для разных типов пользователей (новичок / опытный / только rules / с нуля)

---

## Что изменилось после первой версии

После первого ревью стало понятно, что репозиторий был слишком «технически правильным» и слишком мало «человечным».

Было добавлено и улучшено:
- `docs/00-start-here.md` — стал точкой входа с живым объяснением, а не просто списком шагов
- `docs/13-install-and-adapt.md` — практическая инструкция по переносу системы к себе
- `docs/14-validation-checklists.md` — чеклисты для проверки каждого компонента
- `docs/15-example-prompts-and-calls.md` — реальные примеры того, как ставить задачи агенту
- `examples/prompts/smoke-tests.md` — короткие тесты для проверки работоспособности
- 5 workflows переписаны с нуля: добавлены примеры, ожидаемые результаты, минимальный формат отчёта
- `ai/skills/systematic-debugging` — стал полноценным скиллом с decision tree

## Чем этот репозиторий отличается от официальной документации

Официальная документация Cline объясняет **что есть**.
Этот репозиторий объясняет **как использовать** — в реальных условиях, с реальными задачами.

| Официальная дока | Этот репозиторий |
|---|---|
| Объясняет механику | Объясняет стратегию |
| Пишет для всех | Пишет для практиков |
| Показывает возможности | Показывает рабочий минимум |
| Нейтральный тон | Голос клуба ProAiCommunity |

---

## Принципы, которые легли в основу

**Меньше — лучше.**
Перегруженная система работает хуже простой. Минимальный рабочий набор лучше полного хаоса.

**Понимание важнее копирования.**
Если ты не понимаешь зачем что-то — не ставь. Лучше разберись, потом добавишь.

**Система должна быть адаптируемой.**
Никакой жёсткой привязки к инфраструктуре. Каждый адаптирует под себя.

**Сначала проверь — потом расширяй.**
Каждый новый компонент проходит smoke test, прежде чем считаться «настроенным».

---

## Что адаптируешь ты

Когда берёшь этот репозиторий — тебе нужно заменить:

- все `YOUR_BOT_TOKEN`, `YOUR_CHAT_ID` и подобные плейсхолдеры
- env-переменные под свои сервисы
- пути к файлам, если они отличаются
- platform-specific команды деплоя
- список MCP и skills под свои реальные задачи

Структура — твоя. Наполнение — тоже твоё.
