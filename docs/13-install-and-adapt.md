## Зачем нужен этот файл

Этот файл отвечает на практический вопрос:
**как перенести public repo в свой setup без слепого копирования чужой системы.**

Здесь главный фокус не на “красивой структуре”, а на трёх вещах:
- что переносить сначала;
- что не переносить буквально;
- как не сломать свой текущий setup.

---

## Главный принцип переноса

Переносить нужно **по слоям**, а не “одним архивом”.

Правильная модель такая:
1. понять, что у тебя уже есть;
2. выбрать минимальный полезный набор;
3. перенести один слой маленьким батчем;
4. проверить, что runtime реально его видит;
5. только потом переходить к следующему слою.

Если слой записан, но не виден — это не перенос, а только запись файла.

---

## Что переносить первым делом

Сначала почти всегда достаточно:
- базовых rules из [`ai/rules/`](../ai/rules/);
- 3–5 нужных skills из [`ai/skills/`](../ai/skills/);
- 2–3 MCP из [`07-mcp-guide-human.md`](07-mcp-guide-human.md);
- hooks — только позже и только при ясной причине.

### Безопасный стартовый порядок
1. Rules
2. Skills
3. MCP
4. Hooks
5. Workflows

---

## Что нельзя копировать буквально

Нельзя переносить как есть:
- placeholders как реальные значения;
- токены, chat id, приватные URL и IP;
- infra-specific команды под чужой сервер;
- private-only MCP без своего use-case;
- planning / handoff артефакты живой рабочей системы;
- n8n JSON без понимания, зачем он нужен.

Правильная модель:
- **структуру** можно переносить;
- **секреты и project-specific значения** подставляются вручную;
- **лишние слои** не переносятся вообще.

---

## Что сделать до первого копирования

Перед переносом полезно:
- сохранить backup текущего setup;
- выписать, какие rules / skills / MCP / hooks уже есть;
- понять 2–3 своих основных сценария работы;
- решить, что у тебя будет global, а что workspace-specific.

Это резко снижает шанс превратить перенос в хаос.

---

## Минимальный сценарий переноса

### Шаг 1. Перенеси базовые rules
Возьми минимальный always-on набор.

### Шаг 2. Добавь 1–2 skills
Не весь стек, а только то, что реально отвечает твоим use-cases.

### Шаг 3. Подключи 2–3 MCP
Например:
- `context7`
- `tavily`
- `playwright` — только если нужен UI / браузер.

### Шаг 4. Прогони smoke test
Пока нет smoke test — не переходи к hooks.

### Шаг 5. Только потом решай, нужны ли hooks
Если их роль не можешь объяснить — пока не ставь.

## Когда создавать свой USER.md / профиль пользователя

После базовых rules полезно добавить пользовательский слой.

Он нужен, чтобы агент понимал:
- кто ты;
- чем занимаешься;
- какой стиль ответа тебе полезен;
- какие задачи для тебя типовые.

За основу можно брать:
- [`../ai/rules/USER_TEMPLATE.md`](../ai/rules/USER_TEMPLATE.md)

Практически безопасная последовательность такая:
1. Rules
2. USER.md / profile layer
3. Skills
4. MCP
5. Hooks при необходимости

## Что проверять после каждого слоя

После каждого батча проверяй отдельно:
1. файл реально записан;
2. файл прочитан обратно;
3. runtime реально видит слой;
4. есть proof через smoke test или видимость в UI.

### Минимальные проверки
- rule влияет на язык / стиль / поведение;
- skill можно явно вызвать;
- MCP реально используется по назначению;
- hook реально срабатывает и не ломает happy-path.

---

## Что делать, если агент пишет «готово», а результата нет

Если setup помогает собирать AI-агент, действуй жёстко:

1. Смотри не только чат, но и интерфейс.
2. Проверяй Rules / Skills / MCP / Configured.
3. Говори конкретно, чего не видно.
4. Не давай агенту переходить дальше, пока текущий шаг не подтверждён.
5. Если слой записан, но не виден — сначала recovery, а не новые массовые записи в другие пути.

Это особенно важно для MCP, skills и hooks.

---

## Типичные ошибки при переносе

- копировать всё сразу;
- не различать rule и skill;
- ставить hooks до базового smoke test;
- тянуть private-only MCP в обычный setup;
- путать Cline workflows и n8n JSON;
- считать “файл существует” доказательством того, что слой уже работает.

---

## Как не сломать свой текущий setup

- не заменяй всё одним махом;
- вноси изменения маленькими батчами;
- после каждого батча делай smoke test;
- держи rollback path: backup, git diff или отдельную копию старого setup.

Если после одного нового компонента всё стало хуже — откатывай именно его, а не продолжай наращивать хаос.

---

## Как понять, что адаптация прошла хорошо

Хороший перенос выглядит так:
- агент стал предсказуемее;
- setup стал понятнее, а не запутаннее;
- ты можешь объяснить назначение каждого перенесённого слоя;
- smoke tests подтверждают поведение;
- новые слои не висят “мёртвыми файлами”.

---

## Мини-чеклист переноса

- [ ] Я сделал backup или понимаю, как откатиться
- [ ] Я переношу систему по слоям, а не всё сразу
- [ ] Я проверяю каждый слой отдельно
- [ ] Я не копирую приватные значения буквально
- [ ] Я могу объяснить, зачем нужен каждый перенесённый компонент
- [ ] Я не иду дальше, пока текущий слой не подтверждён
