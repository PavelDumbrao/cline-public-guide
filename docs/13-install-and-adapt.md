## Зачем этот файл
Этот файл отвечает на практический вопрос:
**как перенести этот repo в свой setup без тупняка и без слепого копирования чужой системы.**

## 1. Что копировать первым делом
Сначала копируй только:
- [`ai/rules/`](../ai/rules/)
- 3–5 нужных элементов из [`ai/skills/`](../ai/skills/)
- 2–3 MCP из [`docs/07-mcp-guide-human.md`](07-mcp-guide-human.md)
- hooks — только потом и только если ты уже понял, зачем они тебе нужны

## 2. Что не копировать буквально
Нельзя копировать буквально:
- placeholders как реальные значения
- bot-specific настройки
- любые команды, если не понимаешь зачем они
- infra-specific части под чужой сервер
- n8n JSON без понимания, как они работают

## 3. Куда класть rules
Обычно rules кладут туда, где их будет читать твой Cline setup.

**Важно:** проверь свою версию Cline.
Пути и способ подключения могут отличаться.

> Для нормальной cross-platform установки теперь обязательно смотри [`docs/25-cross-platform-installation-paths.md`](25-cross-platform-installation-paths.md). Там логика не только про пути, но и про read-back, visibility check и recovery, если слой положен не туда.

Если у тебя уже есть текущая система rules:
- не смешивай сразу всё подряд
- добавляй по 1–2 правила и проверяй поведение

## 4. Куда класть skills
Обычно skills лежат в директории skills, где Cline их видит как отдельные папки с `SKILL.md`.

**Это шаблон, адаптируй под свой setup.**

Правило:
- 1 папка = 1 skill
- внутри должен быть `SKILL.md`

## 4A. Когда создавать свой USER.md / профиль пользователя
До расширения setup полезно создать свой пользовательский слой.

Это нужно, чтобы агент понимал:
- кто ты;
- чем ты занимаешься;
- как тебе удобнее получать результат;
- что для тебя важно в работе.

За основу бери шаблон: [`../ai/rules/USER_TEMPLATE.md`](../ai/rules/USER_TEMPLATE.md)

Минимальная логика такая:
1. Сначала rules.
2. Потом короткий USER.md / профиль пользователя.
3. Потом skills и MCP.

Так агент быстрее понимает, как с тобой работать, и меньше уходит в абстракцию.

## 5. Куда класть hooks
Hooks клади только туда, где их реально читает твой Cline.

**Проверь свою версию Cline.**

На старте лучше использовать только:
- `TaskStart`
- `PreToolUse`

## 6. Как работать с workflows
Здесь есть **Cline markdown workflows** в `ai/workflows/`.
Они нужны как reusable procedures.

И отдельно есть **n8n JSON** в `examples/n8n/`.
Это не одно и то же.

## 7. Что заменить вручную
Обязательно вручную заменить:
- `<API_KEY>`
- `<BOT_TOKEN>`
- `<CHAT_ID>`
- `<PRIVATE_URL>`
- `<PRIVATE_IP>`
- любые env placeholders
- project-specific названия

## 8. Как проверить, что Cline это видит
Проверяй по шагам:
- агент придерживается правил языка/стиля
- нужный skill можно явно вызвать
- hook реально срабатывает на событии
- MCP реально используется по назначению

## 9. Типичные ошибки при переносе
- копировать всё сразу
- не понимать разницу между rule и skill
- ставить hooks до базового smoke test
- тянуть private-only MCP в обычный setup
- путать Cline workflows и n8n JSON

## 10. Минимальный smoke test после переноса
1. Проверь rule по языку и стилю
2. Проверь docs search
3. Проверь вызов одного skill
4. Проверь безопасное поведение команды
5. Если есть playwright — проверь простое открытие страницы

## 11. Практические варианты layout

| Что | Частый глобальный вариант | Частый project/workspace вариант | Комментарий |
|---|---|---|---|
| rules | Отдельная глобальная директория правил | `.clinerules/` или локальная project-директория | Часто встречается по-разному, уточни по своей версии Cline |
| skills | Глобальная директория skills | `.agents/skills/` внутри проекта или workspace | У разных setup'ов может отличаться способ подключения |
| hooks | Глобальная директория hooks | project-specific hooks рядом с проектом | Часто встречается так, но зависит от способа установки |
| workflows | Отдельная папка markdown workflows | Локальная project/workspace папка workflows | Важно не путать их с n8n JSON |

## 12. Быстрый сценарий переноса
1. Возьми минимальный набор rules
2. Положи rules в свой layout
3. Положи 1–2 skills
4. Добавь 2–3 MCP
5. Перезапусти Cline или заставь его перечитать setup
6. Сделай smoke test на rules
7. Сделай smoke test на skills
8. Сделай smoke test на MCP
9. Если всё ок — только потом, при реальной необходимости, подключай hooks

## 13. Что делать, если Cline “не видит” rules/skills/hooks
Типовые причины:
- не та папка
- не тот формат файла
- у skill нет `SKILL.md`
- hook не executable
- конфликт старого и нового layout
- Cline не был перезапущен / не перечитал конфиг
- файл создан, но не подключён в твоём способе setup

## 13A. Что делать, если агент пишет «готово», а результата нет
Если агент пишет, что всё установлено, а ты этого не видишь, действуй так:

1. Сначала проверь интерфейс, а не только чат:
   - Rules
   - Skills
   - MCP
   - Configured
2. Потом скажи агенту конкретно, чего именно ты не видишь.
3. Попроси его не идти дальше, пока текущий шаг не подтверждён.
4. Если речь про MCP — попроси проверить, нужен ли reload или reopen окна.
5. Если речь про skill или rule — попроси перепроверить путь и формат, а не спорить словами.
6. Если нужно — попроси его сходить в официальную документацию и перепроверить схему настройки.

Не каждая промежуточная ошибка означает провал.
Провалом считается не ошибка в процессе, а ситуация, когда в интерфейсе так и не появился рабочий результат.

## Практическое правило
Если что-то “не видно”, не добавляй ещё 10 новых файлов.
Сначала проверь один компонент и добейся, чтобы он реально заработал.

## 14. Правильный порядок переноса системы
Самая безопасная последовательность такая:

1. Сначала audit текущего setup.
2. Потом минимальный перенос rules.
3. Потом 1–2 нужных skills.
4. Потом 2–3 MCP.
5. Только потом hooks.
6. Затем workflows, если уже есть повторяемые процедуры.

Это защищает тебя от ситуации, когда непонятно, что именно сломалось.

## 15. Что стоит сделать до первого копирования
Перед переносом полезно:
- сохранить backup текущих настроек;
- зафиксировать, какие rules/skills/hooks/MCP уже есть;
- понять, какие из них реально используются;
- выписать 2–3 своих основных сценария работы.

Тогда ты будешь адаптировать систему под задачу, а не просто копировать красивую структуру.

## 16. Копировать как есть / адаптировать / не переносить

| Что | Как переносить |
|---|---|
| Общие rules по языку, поиску, качеству | Обычно адаптировать минимально |
| Skills по зрелым повторяемым задачам | Переносить выборочно |
| Hooks | Ставить только после понимания и smoke tests |
| MCP | Подключать по одному |
| Приватные URL, токены, IP, chat ID | Не переносить буквально |
| Личные project notes автора | Не копировать как часть публичного setup |

## 17. Как не сломать свой текущий setup
- не заменяй всё одним махом;
- вноси изменения маленькими батчами;
- после каждого батча делай smoke test;
- храни rollback path: backup, git diff или отдельную копию старых файлов.

Если после добавления одного компонента всё стало хуже — откатывай именно этот компонент, а не продолжай наращивать хаос.

## 18. Что делать после успешного переноса
Когда базовый перенос заработал:
- зафиксируй, что именно у тебя реально используется;
- убери лишнее;
- запиши project-specific решения отдельно;
- не тащи в public template свои секреты и приватные детали;
- расширяй систему только под повторяющиеся задачи.

## 19. Как понять, что адаптация прошла хорошо
Хороший перенос выглядит так:
- агент говорит и действует предсказуемее;
- setup стал понятнее, а не запутаннее;
- ты можешь объяснить назначение каждого слоя;
- у тебя нет ощущения, что система живёт своей жизнью;
- smoke tests реально подтверждают поведение.

## 20. Типичный провал при адаптации
Самый частый провал — копировать не систему, а внешний вид системы.

То есть человек переносит:
- кучу файлов;
- сложную структуру;
- лишние MCP;
- непонятные hooks;

но при этом не понимает:
- что у него должно работать always-on;
- что должно включаться по запросу;
- что автоматизировано системой;
- что требует его ручной замены.

Именно поэтому перенос нужно делать по слоям, а не «одним архивом».
