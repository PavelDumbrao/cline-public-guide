## Что такое Cline Workflow

Cline Workflow — это markdown-процесс для повторяемой процедуры.

Это:
- не bash-скрипт;
- не hook;
- не skill;
- не n8n JSON.

Workflow отвечает на вопрос:
**как пройти по повторяемому процессу без пропусков и без “телепатии”?**

---

## Когда workflow реально нужен

Workflow оправдан, если процедура:
- повторяется;
- состоит из нескольких шагов;
- имеет точки проверки;
- на ней легко забыть важные действия;
- должна выполняться одинаково и предсказуемо.

Примеры хороших workflow:
- аудит setup перед переносом;
- проверка rules / skills / hooks после установки;
- подготовка публичной версии приватного repo;
- review lesson candidate после исправленной ошибки.

---

## Когда workflow НЕ нужен

Не создавай workflow, если:
- задача одноразовая;
- это просто знания по теме, а не процесс;
- это should-always-run policy;
- это уже лучше решается hook’ом;
- процесс ещё слишком сырой и постоянно меняется.

---

## Workflow vs Skill vs Hook vs Rule

| Если тебе нужно... | Что выбрать |
|---|---|
| Always-on поведение | **Rule** |
| Экспертизу по теме | **Skill** |
| Автоматическую реакцию на событие | **Hook** |
| Повторяемую многошаговую процедуру | **Workflow** |

### Самая частая путаница

#### Workflow vs Skill
- **Skill** даёт доменную экспертизу;
- **Workflow** даёт порядок шагов и quality gates.

#### Workflow vs Hook
- **Hook** срабатывает автоматически по событию;
- **Workflow** проходит человек или агент как процедуру.

Если нужен принцип **«всегда срабатывает сам»** — это скорее hook.
Если нужен принцип **«пройди вот эту процедуру шаг за шагом»** — это workflow.

#### Workflow vs n8n workflow
- **Cline Workflow** — markdown-процесс для агента;
- **n8n workflow JSON** — импортируемый граф для n8n.

Это принципиально разные вещи.

---

## Как выглядит хороший workflow

Хороший workflow обычно содержит:
- цель;
- входные условия;
- ограничения;
- последовательность шагов;
- точки проверки;
- критерий завершения;
- типичные ошибки.

То есть workflow — это не просто “сделай 1, 2, 3”,
а повторяемая процедура с понятными gates.

---

## Мини-шаблон мышления для workflow

Когда проектируешь workflow, ответь на 6 вопросов:

1. Что запускает этот процесс?
2. Что должно быть на входе?
3. Какие шаги обязательны?
4. Что нужно проверить после ключевых шагов?
5. Когда процесс считается завершённым?
6. Какие ошибки здесь типичны?

Если на эти вопросы нет внятного ответа — возможно, workflow ещё рано делать.

---

## Как workflow связан с остальной системой

- **Rules** задают фоновые ограничения.
- **Skills** помогают глубоко пройти отдельные шаги.
- **Hooks** иногда автоматизируют часть проверки или защиты.
- **MCP** дают внешние инструменты, если шаг этого требует.

То есть workflow почти никогда не живёт изолированно.
Он собирает остальные слои в один процесс.

---

## Как проверять, что workflow хороший

Проверяй не факт существования файла, а полезность:
- по нему можно пройти без догадок?
- он уменьшает хаос?
- он экономит время?
- он не дублирует existing rules / skills / hooks?
- он не устарел относительно текущего setup?

### Мини-чеклист review
- [ ] workflow описывает именно процесс
- [ ] в нём есть шаги и проверки
- [ ] он не заменяет rule там, где нужен always-on policy
- [ ] он не подменяет hook там, где нужна event-автоматика
- [ ] по нему можно пройти без телепатии

---

## Типичные ошибки

### Ошибка 1
Называть workflow любой markdown-файл.

### Ошибка 2
Делать workflow на одноразовую задачу.

### Ошибка 3
Пытаться заменить workflow’ом hook.

### Ошибка 4
Пытаться заменить workflow’ом skill.

### Ошибка 5
Писать слишком общий workflow, который ничего не помогает сделать на практике.
