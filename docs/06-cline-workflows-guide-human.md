# 06. Cline Workflows Guide

## Что такое Cline Workflow
Cline Workflow — это markdown-файл, который описывает **процесс работы самого Cline**.

Это не bash-скрипт.
Это не hook.
Это не n8n JSON.

## Зачем нужны Cline Workflows
Они полезны, когда задача повторяется:
- установить новый skill
- проверить rules
- подготовить публичный репозиторий
- провести аудит setup

## Где они лежат в этом репозитории
- `ai/workflows/`

## Чем workflow отличается от skill
- skill = знания и правила по теме
- workflow = последовательность шагов

## Чем workflow отличается от n8n workflow JSON
- Cline Workflow — markdown-инструкция для AI
- n8n workflow JSON — импортируемый граф для n8n

## Когда они нужны
Используй workflow, если хочешь:
- стандартизировать повторяющуюся процедуру
- уменьшить число забытых шагов
- ускорить рутинные задачи

## Типичная ошибка
Назвать n8n JSON «workflow для Cline».
Это неправильно и создаёт путаницу у новичка.

## Когда workflow действительно оправдан
Workflow нужен, если процедура:
- повторяется;
- состоит из нескольких шагов;
- на ней часто забывают важные проверки;
- её хочется делать одинаково и предсказуемо.

Примеры зрелых workflow:
- аудит setup перед переносом в новый проект;
- проверка rules / skills / hooks после установки;
- подготовка публичной версии приватного repo;
- безопасное подключение нового MCP;
- review lesson candidate после исправленной ошибки.

## Когда workflow НЕ нужен
Не создавай workflow, если:
- задача одноразовая;
- это просто набор знаний по теме, а не последовательность шагов;
- это должно работать always-on;
- это уже лучше решается hook'ом;
- процесс ещё слишком сырой и постоянно меняется.

## Как выглядит хороший workflow
Хороший workflow обычно содержит:
- цель;
- входные данные;
- ограничения;
- последовательность шагов;
- точки проверки;
- критерий завершения;
- типичные ошибки.

То есть workflow — это не просто список «сделай 1, 2, 3»,
а понятная повторяемая процедура с quality gates.

## Как workflow связан с rules, skills, hooks и MCP
- **Rules** задают фоновые ограничения и политику.
- **Skills** дают экспертизу внутри отдельных шагов workflow.
- **Hooks** могут автоматизировать часть проверки или защиты.
- **MCP** дают внешние инструменты, если они нужны конкретным шагам.

Поэтому workflow почти никогда не живёт изолированно.
Он собирает остальные слои в один повторяемый процесс.

## Мини-шаблон хорошего workflow
Можно думать о workflow так:

1. Что запускает этот процесс?
2. Что должно быть на входе?
3. Какие шаги идут по порядку?
4. Что нужно проверить после каждого шага?
5. Когда процесс считается завершённым?
6. Какие ошибки типичны и как их заметить?

## Как проверять workflow
Проверяй не только «существует ли markdown-файл», а:
- проходит ли человек или агент по нему без пропусков;
- уменьшает ли он хаос;
- реально ли он экономит время;
- не устарел ли он относительно текущего setup.

## Мини-чеклист для workflow review
- [ ] workflow описывает повторяемый процесс, а не просто тему
- [ ] в нём есть шаги и точки проверки
- [ ] он не дублирует skill и не пытается заменить rules
- [ ] по нему можно пройти без телепатии
- [ ] он помогает не забывать критичные действия
