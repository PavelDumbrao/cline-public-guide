# 15. Example Prompts and Calls

- [`03-rules-guide-human.md`](03-rules-guide-human.md) — если хочешь лучше понимать, что именно проверяют prompt-примеры

## A. Как проверить rule

### Примеры
- «Ответь по-русски и дай краткий план из 3 шагов по настройке README.»
- «Если не уверен в API, сначала найди официальную документацию и только потом предлагай код.»
- «Проверь эту задачу с приоритетом безопасности и не предлагай опасные команды без объяснения.»
- «Сначала изучи текущую структуру проекта, потом предложи минимальный план изменений.»

## B. Как явно вызвать skill

### systematic-debugging
- «Используй systematic-debugging и разложи баг 500 ошибки по гипотезам.»
- «Применяй systematic-debugging: сначала воспроизведение, потом гипотезы, потом проверка.»

### frontend-design
- «Используй frontend-design и предложи структуру лендинга с понятной иерархией блоков.»
- «Применяй frontend-design: нужен адаптивный экран входа без перегруза.»

### deployment-guide
- «Используй deployment-guide и предложи безопасный чеклист деплоя этого проекта.»
- «Примени deployment-guide: хочу понять, как проверить env и healthcheck перед релизом.»

### local-code-search
- «Используй local-code-search и найди, где в проекте реализована авторизация.»
- «Примени local-code-search: найди смыслом место, где обрабатываются retries HTTP.»

### n8n-http-request
- «Используй n8n-http-request и помоги настроить POST запрос с JSON body и headers.»
- «Примени n8n-http-request: нужно проверить auth, body и expected response для этой ноды.»

## C. Как использовать workflow

### /audit-setup
- «Запусти логику `/audit-setup` для этого проекта и покажи, что здесь reusable, а что private-only.»

### /install-skill
- «Пройди по workflow `/install-skill`: хочу добавить новый skill для smoke test.»

### /review-rules
- «Используй `/review-rules` и скажи, какие rules у меня лишние или слишком личные.»

### /prepare-public-repo
- «Примени `/prepare-public-repo`: нужно превратить текущий setup в безопасный публичный шаблон.»

### /sanitize-private-config
- «Иди по `/sanitize-private-config` и скажи, что в этих файлах надо заменить placeholder-ами.»

## D. Как попросить Cline сделать self-check
- «Сделай self-check: что ты изменил, что ещё надо проверить вручную, где риски?»
- «Проведи критический review своего результата: что может быть сломано?»
- «Проверь себя как новичок: что в твоём ответе всё ещё непонятно?»
- «Сделай финальную проверку на секреты, путаницу сущностей и слабые места.»

## E. Плохие запросы vs хорошие запросы

### Слишком расплывчато
**Плохо:** «Сделай мне setup.»
**Хорошо:** «Собери минимальный Cline setup: 5 rules, 3 skills, 3 MCP и без hooks.»

### Слишком много сразу
**Плохо:** «Настрой всё: rules, skills, hooks, MCP, n8n, Telegram, GitHub, браузер, деплой.»
**Хорошо:** «Сначала собери базовый setup. Потом отдельно добавим hooks. Потом отдельно MCP.»

### Без критерия успеха
**Плохо:** «Проверь проект.»
**Хорошо:** «Проверь проект: есть ли секреты, private-only файлы и путаница между Cline workflows и n8n JSON.»

### Без пошаговости
**Плохо:** «Переделай всё правильно.»
**Хорошо:** «Сначала аудит. Потом список проблем. Потом правки по одному блоку.»

## F. Как попросить выбрать модель под задачу
- «Подбери стратегию моделей: что отдать дешёвой рабочей модели, а что — сильной reasoning-модели.»
- «Скажи, нужна ли здесь сильная ревью-модель или хватит рабочей дешёвой.»

## G. Как попросить сделать compact / handoff
- «Сделай короткий handoff summary: что сделано, что осталось, какие риски.»
- «Сократи контекст до минимально нужного summary для следующей сессии.»

## H. Как попросить безопасно использовать browser automation
- «Используй browser automation в read-only режиме: открой страницу, проверь консоль и сеть, ничего не кликай без необходимости.»
- «Скажи, нужен ли здесь browser automation вообще, или можно обойтись без него.»

## I. Как попросить собрать skill из документации
- «Изучи эту документацию и скажи, стоит ли делать из неё отдельный skill, а не просто note.»
- «Пройди по workflow `/create-skill-from-docs` и предложи reusable skill structure.»

## J. Как попросить восстановиться после model drift
- «Похоже, модель ушла в drift. Сначала диагностируй симптом, потом предложи recovery plan.»
- «Примени `/recover-from-model-drift`: что нужно переключить — модель, контекст или сессию?»

## K. Как попросить своего Cline изучить этот репозиторий и собрать setup

### Короткий вариант
**Промпт:**
```text
Изучи этот репозиторий как учебный шаблон по настройке Cline:
https://github.com/PavelDumbrao/cline-public-guide

Сначала сделай аудит. Потом предложи минимальный setup под мои задачи. Потом по шагам помоги мне создать rules, skills, MCP и при необходимости hooks. После каждого этапа проверяй результат.
```

### Нормальный рабочий вариант
**Промпт:**
```text
Изучи этот репозиторий как учебный шаблон по настройке Cline:
https://github.com/PavelDumbrao/cline-public-guide

Действуй по шагам.

Что мне нужно:
1. Сначала проанализируй repo и объясни, какие rules, skills, hooks и MCP мне реально нужны для старта.
2. Потом предложи минимальный setup именно под мои задачи.
3. Затем помоги по шагам создать или адаптировать для меня:
   - rules
   - skills
   - hooks (если реально нужны)
   - MCP
4. Не копируй чужую систему 1-в-1 — адаптируй под мой проект.
5. Если нужны мои значения, оставляй placeholders и отдельно говори, что заменить вручную.
6. После каждого этапа делай проверку, а не просто говори «готово».

Начни с аудита и короткого плана.
```

### Важная формулировка для ученика
Если хочешь, чтобы Cline **не просто посмотрел repo**, а реально собрал тебе систему, важно писать не так:

**Плохо:**
`Посмотри этот репозиторий.`

А так:

**Хорошо:**
`Изучи этот репозиторий как шаблон и по шагам помоги мне создать свой рабочий setup: rules, skills, MCP и hooks, с проверкой после каждого этапа.`


## L. Как попросить AI безопасно установить setup по этому repo

### Строгий вариант
- «Используй mega-prompt установщика из [`docs/26-mega-prompt-for-ai-installer.md`](26-mega-prompt-for-ai-installer.md) и работай только слоями с visibility check после каждого батча.»
- «Сначала определи ОС и runtime, потом реши global vs workspace, потом ставь layers маленькими батчами и проверяй, что Cline реально их видит.»

### Если есть риск wrong-path кейса
- «Перед записью файлов используй path-логику из [`docs/25-cross-platform-installation-paths.md`](25-cross-platform-installation-paths.md). Если не уверен в пути — остановись и уточни.»
- «Если слой уже записан, но не видно, не продолжай. Используй recovery flow из [`docs/27-setup-self-check-and-recovery.md`](27-setup-self-check-and-recovery.md).»
