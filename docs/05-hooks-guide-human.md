# 05. Hooks Guide

## Что такое hook
Hook — это скрипт, который запускается на событии Cline.

Примеры событий:
- старт задачи
- завершение задачи
- до запуска инструмента
- после запуска инструмента

## Чем hook отличается от rule
- rule влияет на поведение AI
- hook выполняет код автоматически

## Когда hooks реально полезны
- запретить опасные команды
- добавить логирование
- добавить напоминание в контекст
- вести телеметрию

## Безопасный стартовый набор
- `TaskStart`
- `PreToolUse`
- `PostToolUse`

## Что не ставить сразу
- hooks с внешними API и токенами
- hooks, которые могут сломать рабочий процесс
- hooks с жёсткой привязкой к инфраструктуре

## Типичная ошибка
Сделать hook «умным», но непрозрачным.
Если hook сложный, новичок быстро перестаёт понимать, что происходит.

## Как проверить hook
1. Сымитируй событие
2. Проверь JSON output
3. Проверь, что stdout содержит только валидный JSON
4. Проверь, что секреты не захардкожены


## Hooks не должны быть монстром
Частая ошибка — пытаться запихнуть всю магию в один hook.

Так делать не надо.

Гораздо устойчивее схема:
- один hook = одна роль;
- общая логика живёт в helper script;
- hooks остаются тонкими и понятными.

Практический пример зрелой архитектуры:
- `PreToolUse` = блокировки и soft guards;
- `PostToolUse` = telemetry;
- `PreCompact` = checkpoint перед compaction;
- `SessionStart` = restore после compaction.

## Hooks и context protection
Hooks особенно полезны, когда нужно защитить контекст от деградации:
- заранее блокировать giant-input кейсы;
- логировать размер и риск tool results;
- сохранять recovery bundle перед compaction;
- восстанавливать continuity после compaction.

Это уже не “красивый бонус”, а реальный production-паттерн для длинных сессий.

## Что не ставить сразу — дополнение
- hooks, которые делают слишком много сразу и которые ты не сможешь отладить

## Safe smoke tests для hooks
Перед тем как считать hook рабочим, прогоняй минимум:
- happy-path сценарий;
- блокирующий сценарий;
- no-regression сценарий, чтобы убедиться, что старый workflow не сломан.

Примеры:
- `git add .env` должен блокироваться;
- `sudo ls /tmp` должен давать warning, а не жёсткий block;
- обычный маленький `read_file` не должен ломаться;
- recovery после compaction должен подниматься без дублей.

## Как понять, что hook уже слишком сложный
- ты не можешь быстро объяснить его роль одной фразой;
- он и блокирует, и логирует, и роутит, и чинит всё подряд;
- его невозможно безопасно тестировать малыми сценариями;
- его проще выключить, чем понять.
