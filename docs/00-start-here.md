## Что такое Cline — коротко и честно

Cline — это не просто AI-чат внутри VS Code.

Это агент, который умеет:
- читать файлы;
- менять код;
- запускать команды;
- пользоваться браузером;
- обращаться к внешним инструментам.

То есть Cline не только **объясняет**, но и **действует**.

Именно поэтому ему нужна настройка.
Без неё поведение будет хаотичным:
- где-то он угадает;
- где-то полезет не туда;
- где-то начнёт работать без ясных ограничений.

## Четыре сущности, которые надо понять сразу

### Rules
Это always-on слой.
Rule отвечает на вопрос:
**как агент должен вести себя всегда?**

Примеры:
- язык ответа;
- честность;
- tool usage;
- поиск документации;
- базовая безопасность.

### Skills
Это on-demand экспертиза.
Skill отвечает на вопрос:
**как агент должен действовать внутри конкретной темы?**

Примеры:
- debugging;
- frontend;
- deployment;
- n8n;
- local code search.

### Hooks
Это автоматика по событиям.
Hook отвечает на вопрос:
**что должно происходить автоматически на старте, завершении, перед командой и т.д.?**

### MCP
Это внешние инструменты и данные.
MCP отвечает на вопрос:
**какие внешние возможности агент может получить извне?**

### ✅ Чек-лист
- [ ] Я понимаю разницу между rules, skills, hooks и MCP
- [ ] Я понимаю, что работает always-on, а что подключается по ситуации
- [ ] Я не путаю инструкции агента и внешние инструменты

## Главная ошибка новичка

> Не копируй чужую живую систему 1-в-1.

Даже если setup выглядит круто, у другого человека:
- свой стек;
- свои сервисы;
- свои токены;
- свои привычки;
- свои project-specific правила.

Если копировать вслепую — получится чужая система в твоём окружении.
Она будет выглядеть похоже, но работать хуже, чем ты ожидаешь.

Правильная модель:
1. понять слои;
2. взять минимальный полезный набор;
3. адаптировать под себя;
4. проверить результат.

## Минимальный стартовый набор

На старте не надо ставить всё.

### Минимум по слоям

**Rules:**
- language/style;
- documentation search;
- search tools;
- code quality;
- tool usage;
- skills management.

**Skills:**
- `systematic-debugging`
- `deployment-guide`
- `local-code-search`
- `frontend-design` — только если нужен UI

**MCP:**
- `context7`
- `tavily`
- `playwright` — только если реально нужен браузер/UI

**Hooks:**
- можно не ставить на первом этапе вообще

### Почему так
- rules снижают хаос;
- skills добавляют экспертизу по запросу;
- MCP расширяют возможности только там, где реально нужны;
- hooks лучше подключать позже, когда уже понятен базовый runtime.

### ✅ Чек-лист
- [ ] Я начинаю с минимума, а не с полного стека
- [ ] Я не тащу hooks слишком рано
- [ ] Я понимаю, зачем нужен каждый компонент стартового набора

## Первый рабочий цикл

Вот самый безопасный порядок:

1. Прочитать базу.
2. Перенести минимальные rules.
3. Добавить 2–4 skills.
4. Подключить 2–3 MCP.
5. Прогнать smoke tests.
6. Только потом решать, нужны ли hooks.

Если setup помогает собирать AI-агент:
- сначала дай ему install-протокол,
- а не просьбу «настрой мне всё».
