# 00. Start Here

## Что такое Cline в одной фразе
Cline — это AI-исполнитель, который умеет читать код, менять файлы, запускать команды, пользоваться браузером и работать по правилам.

## С чего начать новичку

### Шаг 1. Понять базовую модель
Сначала разберись в 4 сущностях:
- rules
- skills
- hooks
- MCP

### Шаг 2. Не копируй чужую систему вслепую
Главная ошибка новичка — взять живую систему автора и перенести её один в один.
Так делать нельзя, потому что:
- у автора свои сервисы
- свои ключи
- свои процессы
- свои привычки

### Шаг 3. Возьми минимум
Стартовый набор:
- 5–7 базовых rules
- 3–5 skills
- 0–3 hooks
- 2–3 MCP

### Шаг 4. Проверь что всё реально работает
После каждой настройки задавай себе 4 вопроса:
1. Зачем это нужно?
2. Когда это используется?
3. Какая типичная ошибка?
4. Как я проверю, что это реально работает?

## Первый запуск за 15 минут

### 1. Какие 5–7 rules взять сначала
Возьми сначала только это:
- `ai/rules/01-core-language-style.md`
- `ai/rules/02-documentation-search.md`
- `ai/rules/05-search-tools.md`
- `ai/rules/06-code-quality.md`
- `ai/rules/07-tool-usage.md`
- `ai/rules/08-skills-management.md`
- `ai/rules/SOUL_TEMPLATE.md`

### 2. Какие 3–4 skills взять сначала
На старте достаточно:
- `systematic-debugging`
- `frontend-design`
- `deployment-guide`
- `local-code-search`

### 3. Какие 2–3 MCP поставить сначала
Минимум:
- `context7`
- `tavily`
- `playwright`

### 4. Какие hooks можно не ставить на первом этапе
На первом этапе можно вообще **не ставить hooks**.
Если очень хочется — ставь только:
- `TaskStart`
- `PreToolUse`

`PostToolUse` и любые внешние уведомления лучше добавить позже.

### 5. Как проверить, что setup реально работает
Сделай мини-проверку:
- агент отвечает на нужном языке
- агент использует структуру из rules
- агент умеет искать docs через MCP
- агент не предлагает опасные действия без проверки
- browser automation реально открывает страницу и видит её

## Первая проверка

Дай Cline 5 коротких задач:

### 1. Проверка языка и стиля
«Ответь по-русски и предложи краткий план из 3 шагов для создания README.»

### 2. Проверка поиска документации
«Найди в официальной документации, как работает `fetch` timeout / abort в JavaScript.»

### 3. Проверка безопасной работы с командами
«Проверь, что произойдёт, если в проекте попробовать `git add .env`, и объясни безопасно без выполнения опасного действия.»

### 4. Проверка skill activation
«Используй systematic-debugging и покажи, как бы ты расследовал баг с 500 ошибкой API.»

### 5. Проверка browser / MCP
«Открой страницу в браузере и проверь, нет ли ошибок в консоли.»

## Когда НЕ расширять setup
Не расширяй setup, если:
- ты не понимаешь, зачем новый компонент нужен
- ты не прошёл базовый smoke test
- ты хочешь просто скопировать чужую систему 1-в-1
- ты не можешь объяснить, что именно улучшит новый MCP / skill / hook

## Что НЕ нужно ставить сразу
- 10+ MCP подряд
- Telegram hooks с токенами
- infra-specific skills
- n8n live MCP без реального n8n use-case

## Главная мысль
Сначала настрой **понятную минимальную систему**, потом расширяй.
