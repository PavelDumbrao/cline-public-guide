> Этот файл нужен, чтобы ученик мог дать своему AI-кодеру **строгий и безопасный установочный промпт**, а не расплывчатое «настрой мне всё».

---

## 1. Зачем нужен отдельный mega-prompt

Обычная ошибка ученика:
- дать агенту слишком широкую задачу;
- не заставить его определить ОС и runtime;
- не разделить global vs workspace;
- не требовать read-back и visibility check;
- не дать stop conditions.

Итог:
- files могли быть записаны «куда-то»;
- агент говорит «готово», хотя ничего реально не подхватилось;
- пользователь не понимает, что проверить дальше.

Этот документ даёт промпт, который заставляет AI-установщика:
- идти по шагам;
- писать слоями;
- проверять видимость после каждого батча;
- останавливаться при низкой уверенности.

---

## 2. Что обязан делать хороший AI-установщик

Перед любыми изменениями он обязан:
1. Определить ОС и тип runtime.
2. Определить global vs workspace scope.
3. Понять, какие layers реально нужны пользователю.
4. Не копировать весь repo механически.
5. Писать маленькими батчами.
6. После каждого батча читать записанные файлы обратно.
7. Проверять, что слой реально **виден** Cline.
8. Останавливаться, если путь, формат или видимость не подтверждены.
9. Не писать следующий батч, пока предыдущий не подтверждён через read-back и visibility check.

## 3. Строгий mega-prompt для AI-установщика

Скопируй и адаптируй этот промпт:

```text
Изучи этот репозиторий как учебный шаблон по настройке Cline:
https://github.com/PavelDumbrao/cline-public-guide

Работай строго по шагам. Не пытайся настроить всё сразу.

Твоя задача:
не просто прочитать repo, а помочь мне безопасно собрать мой рабочий Cline setup.

Критически важно:
- не копируй чужую систему 1-в-1;
- не угадывай пути и runtime;
- не объявляй слой готовым, пока не проверил, что Cline реально его видит;
- если я не вижу результат в интерфейсе, считай, что шаг не завершён;
- запрещено писать следующий батч, пока предыдущий не подтверждён read-back + visibility check;
- если слой не виден, сначала recovery, а не запись новых файлов в другие пути наугад.

Порядок работы:

Шаг 0. Сначала проверь базовую готовность
1. Убедись, что у тебя есть доступ к repo.
2. Проверь, хватает ли тебе MCP для чтения репозитория, документации и поиска.
3. Если тебе нужен GitHub MCP, Tavily, Exa или другой MCP до начала основной работы — сначала помоги мне его поставить.
4. Если нужен токен или логин, не просто проси его, а сначала дай мне прямую ссылку, где это взять.
5. После установки MCP проверь, появился ли он в Configured или в рабочем списке серверов.
6. Если без reload / reopen окна новый MCP не появится — скажи об этом прямо.

Шаг 1. Сначала определи среду
1. Определи ОС: macOS / Windows / Linux / WSL.
2. Определи, где именно работает runtime Cline: local, remote, server-side, VS Code storage path или другой runtime.
3. Отдельно реши, что должно быть global, а что workspace-specific.
4. Если уверенность по пути низкая — остановись и задай короткий точечный вопрос.

Шаг 2. Сделай аудит того, что мне реально нужно
1. Сначала объясни, какие layers мне нужны для старта:
   - rules
   - skills
   - MCP
   - hooks
   - workflows
2. Предложи минимальный setup именно под мои задачи.
3. Не тащи hooks, private-only MCP и лишние skills без необходимости.

Шаг 3. Устанавливай слоями, маленькими батчами
Порядок установки:
1. rules
2. skills
3. MCP
4. hooks
5. workflows

Работай с этой настройкой в одном активном окне.
Не веди один и тот же install-flow одновременно в нескольких окнах.

После каждого батча ты обязан:
- показать, какие файлы созданы или изменены;
- прочитать эти файлы обратно;
- сделать visibility check;
- дать proof, чем именно visibility подтверждена;
- коротко отчитаться, что реально подтверждено.

До следующего батча нельзя переходить, пока ты явно не ответил:
- file written = yes/no
- file read back = yes/no
- runtime visibility confirmed = yes/no
- proof used = какой именно smoke test / prompt / event / tool call это подтвердил

Шаг 4. Правила установки rules
1. Установи все базовые rules по списку из гайда, а не 2–3 тестовых rules.
2. Не копируй все подряд без объяснения, зачем они нужны.
3. После установки перечисли, какие именно rules уже установлены.
4. Попроси меня проверить rules в интерфейсе.
5. Если я не вижу часть rules, не переходи к следующему шагу.
6. После записи rules проверь:
   - язык/стиль;
   - docs-search behavior;
   - tool-usage discipline;
   - базовую честность и предсказуемость поведения.
7. Если rule файл существует, но поведение не изменилось — не считай задачу завершённой.
8. Не переходи к skills, пока весь базовый rule-layer не подтверждён visibility check.
9. После базовых rules помоги мне создать пользовательский слой / профиль пользователя, если его ещё нет.

Шаг 5. Правила установки skills
1. Добавляй только те skills, которые реально нужны под мои use-cases.
2. Убедись, что каждая skill-папка содержит `SKILL.md`.
3. После добавления skill проверь, что его можно явно вызвать.
4. Попроси меня проверить skills в интерфейсе.
5. Если skill не появился в UI, считай, что установка не завершена.
6. Не спорь со мной, если я говорю, что не вижу skill.
7. В таком случае перепроверь путь, формат и место создания.
8. Если надо — создай тестовый skill как маяк, найди реальный рабочий путь и только потом ставь остальные.
9. Не переходи к следующему skill или MCP, пока хотя бы один установленный skill не подтверждён и мной, и UI.

Шаг 6. Правила установки MCP
1. Подключай MCP по одному.
2. Для каждого MCP отдельно объясняй:
   - зачем он нужен;
   - чем он отличается от уже существующих;
   - нужен ли токен/логин;
   - какой у него expected smoke test.
3. Если нужен токен или логин, сначала дай прямую ссылку, где это взять.
4. После добавления MCP проверь, что он реально используется по назначению.
5. Попроси меня проверить MCP в интерфейсе и в Configured.
6. Если MCP записан в config, но не виден — сначала path/runtime audit и проверка reload/reopen, а не новые MCP.
7. Не тащи private-only MCP без явного смысла.

Шаг 7. Правила установки hooks
1. Hooks добавляй только если есть понятная причина.
2. Начинай с простых hooks.
3. Проверяй, что stdout hooks содержит только валидный JSON.
4. Не хардкодь секреты.
5. Если hook ломает workflow — сразу остановись и покажи причину.
6. Не ставь hooks, если базовый setup rules/skills/MCP ещё не подтверждён.

Шаг 8. Правила установки workflows
1. Не путай Cline markdown workflows и n8n JSON.
2. Добавляй workflow только если процедура реально повторяемая.
3. После добавления workflow объясни, для какой именно процедуры он нужен.
4. Не добавляй workflow, если это на самом деле rule, hook или просто markdown-заметка.

Шаг 9. Обязательный self-check после каждого слоя
После каждого слоя отдельно отвечай:
1. Что именно добавлено?
2. Какие файлы реально изменены?
3. Что подтверждено read-back'ом?
4. Что подтверждено visibility check'ом?
5. Какой именно proof использован для visibility check?
6. Что ещё остаётся неподтверждённым?
7. Нужен ли следующий шаг или лучше сначала остановиться?

Используй жёсткий mini-contract:
- file_written: yes/no
- file_read_back: yes/no
- runtime_visibility_confirmed: yes/no
- proof_used: ...
- unresolved_risks: ...

Шаг 10. Stop conditions
Ты обязан остановиться и попросить подтверждение, если:
- не уверен в пути;
- не понимаешь, какой runtime читает конфиг;
- слой записан, но не виден;
- пользователь говорит, что не видит rule / skill / MCP в интерфейсе;
- предыдущий батч не подтверждён read-back и visibility check;
- нужен логин, токен или чувствительное действие;
- задача начала требовать массового копирования без понятной верификации;
- ты собираешься писать один и тот же слой сразу в несколько candidate paths без подтверждённого recovery-плана.

Шаг 11. Placeholder policy
Если нужны мои значения, используй placeholders и явно помечай, что я должен заменить вручную.
Никогда не подставляй вымышленные токены, приватные URL, IP, логины и ключи как будто они реальные.

Шаг 11b. Что НЕ переносить автоматически
Не переносить без отдельного объяснения и подтверждения:
- private-only MCP
- лишние hooks
- все rules подряд без приоритизации
- весь skill-stack автора целиком
- planning/handoff файлы вроде implementation artifacts
- чувствительные project-specific детали без sanitization

Шаг 12. Формат отчёта
После каждого этапа давай короткий отчёт в таком виде:
- что анализировал
- что решил ставить
- что реально записал
- что подтверждено read-back
- что подтверждено visibility check
- что не подтверждено
- какие риски
- какой следующий шаг

Начни с аудита среды, а не с записи файлов.
```

## 4. Короткий safe-вариант промпта

Если не нужен полный длинный промпт, используй короткий вариант:

```text
Изучи этот repo как шаблон настройки Cline и помоги мне безопасно собрать мой setup.

Сначала:
1. Определи ОС и runtime.
2. Реши отдельно global vs workspace scope.
3. Предложи минимальный setup под мои задачи.

Потом устанавливай слоями:
- rules
- skills
- MCP
- hooks
- workflows

После каждого слоя:
- покажи, какие файлы изменены;
- прочитай их обратно;
- проверь, что Cline реально видит слой;
- не говори «готово», если видимость не подтверждена.

Если путь, runtime или видимость неясны — остановись и задай короткий вопрос.
```

## 5. Плохой промпт vs хороший промпт

### Плохо
```text
Настрой мне Cline полностью.
```

Почему плохо:
- нет ОС;
- нет scope;
- нет install order;
- нет self-check;
- нет stop conditions.

### Лучше
```text
Сначала определи ОС и runtime. Потом предложи минимальный setup. Потом устанавливай слоями и после каждого слоя проверяй, что Cline реально его видит. Не переходи к следующему батчу, пока предыдущий не подтверждён.
```

### Хорошо
Используй строгий mega-prompt из этого документа.

## 6. Что должен проверять ученик после ответа AI-установщика

Даже если промпт хороший, ученик не должен верить ответу на слово.

Минимум нужно проверить:
- слой реально записан туда, куда надо;
- файл реально прочитан обратно;
- есть поведенческая проверка, а не только факт существования файла;
- AI не замаскировал неопределённость под уверенность;
- нет лишних слоёв, которые были добавлены «на всякий случай».
