## 1. Зачем вообще нужен этот файл

Большинство ошибок при настройке Cline происходят не потому, что человек “не умеет писать markdown”, а потому что он путает уровни системы:
- что должно жить в rules;
- что должно быть skill;
- где нужен hook;
- где нужен workflow;
- а где вообще достаточно project note.

Этот документ — карта всей системы.

---

## 2. Пять основных сущностей

### Rules
Rules — это always-on уровень.

Они отвечают на вопрос:
> как агент должен вести себя **всегда**?

Примеры:
- язык ответов;
- требования безопасности;
- когда искать документацию;
- как выбирать инструменты;
- как выбирать skills.

### Skills
Skills — это on-demand уровень экспертизы.

Они отвечают на вопрос:
> как агент должен действовать в **конкретной теме**?

Примеры:
- systematic-debugging;
- frontend-design;
- deployment-guide;
- local-code-search;
- n8n-*.

### Hooks
Hooks — это событийная автоматика.

Они отвечают на вопрос:
> что должно происходить **автоматически** на lifecycle-событиях?

Примеры:
- блокировка опасной операции;
- логирование;
- checkpoint перед compaction;
- restore после compaction;
- уведомление о завершении задачи.

### MCP
MCP — это внешние инструменты и источники данных.

Они отвечают на вопрос:
> какие данные и внешние возможности агент может получить извне?

Примеры:
- поиск в интернете;
- документация библиотек;
- браузер;
- GitHub;
- Railway;
- Notion;
- Hostinger API.

### Workflows
Workflows — это повторяемые многошаговые процедуры.

Они отвечают на вопрос:
> как пройти по **процессу**, если задача состоит из повторяемых этапов?

Примеры:
- аудит setup;
- подготовка публичного repo;
- review rules;
- установка нового skill;
- sanitization private config.

---

## 3. Как они связаны между собой

### Rules → Skills
Rules говорят, **когда skill нужен**.
Skill говорит, **как глубоко работать внутри темы**.

То есть:
- Rule = routing
- Skill = expertise

### Rules → Hooks
Rules задают policy.
Hooks превращают policy в автоматическое enforcement.

Пример:
- Rule: не тащи секреты и giant мусор куда попало;
- Hook: блокирует опасный кейс или пишет telemetry.

### Skills ↔ Hooks
Обычно skill и hook не конкурируют.

- Skill нужен, когда нужна глубина и reasoning;
- Hook нужен, когда действие должно происходить автоматически всегда.

### MCP ↔ Skills
Skill может говорить, **какой MCP использовать и в каком порядке**.

Например:
- implementation-research → Context7 → GitHub → Exa → Tavily;
- browser-automation → Playwright MCP;
- local-code-search → semantic search по репо.

### Workflows ↔ всё остальное
Workflow может использовать:
- rules как базовые ограничения;
- skills как экспертизу;
- hooks как автоматизацию;
- MCP как внешние инструменты.

---

## 4. Что работает всегда, а что по ситуации

| Сущность | Always-on? | Кто запускает | Когда включается |
|---|---|---|---|
| Rules | Да | Система | На каждой задаче |
| Skills | Нет | Агент / routing | Когда задача соответствует skill |
| Hooks | Автоматически по событию | Система | На lifecycle event |
| MCP | Нет | Агент | Когда нужен внешний инструмент |
| Workflows | Нет | Агент / человек | Когда запускается повторяемая процедура |

---

## 5. В каком порядке строить систему

### Шаг 1. Rules
Сначала всегда строится базовый policy layer.

Почему:
- без него агент хаотичен;
- без него skills и MCP тоже будут использоваться хаотично.

### Шаг 2. Skills
Потом добавляется on-demand экспертиза.

Почему:
- skills хороши, когда уже понятно, какие темы реально нужны;
- не надо тащить 20 skills сразу, если ты ещё не знаешь свой workflow.

### Шаг 3. MCP
Потом подключаются внешние инструменты.

Почему:
- сначала надо понять, что агент вообще должен делать;
- только потом расширять его “руки и глаза”.

### Шаг 4. Hooks
Потом hooks.

Почему:
- hooks опаснее всего для новичка;
- они автоматические и могут ломать процесс незаметно;
- hooks стоит ставить только когда базовый setup уже понятен.

### Шаг 5. Workflows
Когда всё стабилизировалось, появляются reusable процессы.

---

## 6. Что куда НЕ надо класть

### В Rules НЕ надо класть
- личные токены;
- IP/домены;
- incident logs одного проекта;
- огромные wiki-тексты;
- пошаговые узкие инструкции, которые нужны только иногда.

### В Skills НЕ надо класть
- always-on policy;
- случайные единичные ошибки;
- секреты;
- личную инфраструктуру без обезличивания;
- всё подряд, лишь бы “не забыть”.

### В Hooks НЕ надо класть
- длинную reasoning-логику;
- тяжёлую магию без отладки;
- непрозрачные блокировки;
- секреты в явном виде.

### В Workflows НЕ надо класть
- то, что должно работать всегда;
- то, что уже лучше решается hook’ом;
- чисто справочную документацию.

---

## 7. Как система должна эволюционировать

Правильная схема не такая:

> каждая ошибка = новый rule/skill/hook

А такая:

1. ошибка или неудачный паттерн обнаружен;
2. ошибка исправлена;
3. исправление проверено;
4. причина реально понята;
5. появляется **lesson candidate**;
6. candidate классифицируется как:
   - Rule
   - Skill
   - Hook
   - Workflow
   - project note / docs / memory-bank
7. только потом создаётся или обновляется соответствующий слой.

### Практическое правило
Если паттерн не повторялся и не проверен — не создавай новый артефакт автоматически.

---

## 8. Rule vs Skill vs Hook vs Workflow — decision table

| Если задача про... | Что выбирать |
|---|---|
| Поведение, которое должно работать всегда | Rule |
| Узкую глубокую экспертизу по теме | Skill |
| Автоматическую реакцию на событие | Hook |
| Повторяемый многошаговый процесс | Workflow |
| Локальный факт/инцидент конкретного проекта | project note / docs / memory-bank |

---

## 9. Где студент чаще всего ломается

### Ошибка 1
Пихает всё в Rules.

### Ошибка 2
Создаёт skill на каждый чих.

### Ошибка 3
Ставит hooks до того, как понял базовый setup.

### Ошибка 4
Тащит слишком много MCP.

### Ошибка 5
Путает workflow с n8n JSON и с обычной markdown-заметкой.

---

## 10. Что считать зрелой системой

Система считается зрелой, если:
- есть короткие и понятные Rules;
- Skills не дублируют Rules;
- Hooks решают только automation/enforcement;
- MCP подключены по делу;
- Workflows используются для реально повторяемых процессов;
- новые артефакты появляются только после проверенных уроков.
