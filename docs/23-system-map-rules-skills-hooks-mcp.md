## Зачем вообще нужна такая карта

Большинство проблем при настройке Cline возникают не из-за markdown и не из-за VS Code.
Они возникают потому, что человек путает уровни:
- что должно жить в rules;
- что должно быть skill;
- где нужен hook;
- когда нужен workflow;
- а когда вообще достаточно project note.

Поэтому сначала надо увидеть всю карту, а уже потом расширять setup.

---

## Пять базовых слоёв

| Слой | Главный вопрос | Что делает | Как включается |
|---|---|---|---|
| **Rules** | Как агент должен вести себя всегда? | Always-on policy | На каждой задаче |
| **Skills** | Как агент должен работать внутри конкретной темы? | On-demand экспертиза | Когда задача подходит под skill |
| **Hooks** | Что должно происходить автоматически на событии? | Automation / enforcement | На lifecycle event |
| **MCP** | Какие внешние данные и инструменты агент может получить? | External capabilities | Когда агент вызывает MCP |
| **Workflows** | Как пройти повторяемую процедуру без пропусков? | Reusable process | Когда запускается workflow |

---

## Как думать о каждом слое

### Rules
Rules — это характер и политика агента.
Они нужны для:
- языка;
- честности;
- выбора инструментов;
- качества кода;
- безопасности;
- routing к skills и search tools.

### Skills
Skills — это не “ещё один слой правил”, а пакет глубокой экспертизы по теме.
Они хороши, когда задача:
- повторяется в одной доменной зоне;
- требует особого workflow;
- не должна жить always-on.

### Hooks
Hooks — это не reasoning и не длинные инструкции.
Они нужны, когда действие должно происходить **автоматически**:
- блокировка опасной операции;
- логирование;
- checkpoint перед compaction;
- restore;
- уведомление после завершения задачи.

### MCP
MCP — это не замена rules или skills.
Это внешний инструмент:
- docs lookup;
- web search;
- browser automation;
- GitHub;
- n8n;
- infra APIs.

### Workflows
Workflow нужен для **повторяемого процесса**.
Он не заменяет ни rule, ни skill, ни hook.
Он нужен, когда важен порядок шагов, quality gates и повторяемость.

---

## Как слои связаны между собой

### Rules → Skills
Rules подсказывают, **когда нужен skill**.
Skill отвечает, **как работать глубоко внутри темы**.

### Rules → Hooks
Rules задают policy.
Hooks превращают часть policy в автоматическое enforcement.

### Skills ↔ MCP
Skill может говорить, **какой MCP использовать и в каком порядке**.
Например:
- docs → Context7;
- research → Tavily / Exa;
- browser automation → Playwright.

### Workflows ↔ всё остальное
Workflow собирает остальные слои в повторяемый процесс:
- rules дают baseline;
- skills дают экспертизу;
- MCP дают внешние возможности;
- hooks иногда автоматизируют часть шага.

---

## В каком порядке строить систему

Почти всегда безопаснее так:

1. **Rules**
2. **Skills**
3. **MCP**
4. **Hooks**
5. **Workflows**

Почему именно так:
- без rules агент хаотичен;
- без понятных задач skills и MCP легко разрастаются зря;
- hooks рано подключать опасно, потому что они автоматические;
- workflows полезны только когда уже есть повторяемая процедура.

---

## Быстрая decision table

| Если задача про... | Что выбирать |
|---|---|
| Поведение, которое должно работать всегда | **Rule** |
| Глубокую экспертизу по теме | **Skill** |
| Автоматическую реакцию на событие | **Hook** |
| Повторяемый многошаговый процесс | **Workflow** |
| Внешние данные или сервис | **MCP** |
| Локальный факт конкретного проекта | **project note / docs / memory-bank** |

---

## Что куда НЕ надо класть

### В Rules не надо класть
- секреты;
- личную инфраструктуру;
- узкие одноразовые инструкции;
- incident notes одного проекта.

### В Skills не надо класть
- always-on policy;
- всё подряд “на всякий случай”;
- частные непроверенные случаи.

### В Hooks не надо класть
- длинное reasoning;
- тяжёлую магию без диагностики;
- захардкоженные секреты;
- непрозрачные блокировки.

### В Workflows не надо класть
- то, что должно работать всегда;
- то, что уже лучше решается hook’ом;
- просто справочную теорию без процесса.

## Где студент чаще всего ломается

### Ошибка 1
Пихает почти всё в rules.

### Ошибка 2
Создаёт skill на каждый чих.

### Ошибка 3
Ставит hooks до базового smoke test.

### Ошибка 4
Тащит слишком много MCP и дублирует инструменты.

### Ошибка 5
Путает workflow, hook и markdown-заметку.

### Ошибка 6
Считает “файл записан” доказательством того, что слой уже работает.

---

## Как система должна эволюционировать

Неправильная схема:
> каждая ошибка = новый rule / skill / hook

Правильная схема:
1. ошибка замечена;
2. исправление реально проверено;
3. причина понята;
4. появляется lesson candidate;
5. он классифицируется как Rule / Skill / Hook / Workflow / note;
6. только потом обновляется нужный слой.

---

## Что считать зрелой системой

Система становится зрелой, когда:
- rules короткие и понятные;
- skills не дублируют rules;
- hooks решают только automation / enforcement;
- MCP подключены под реальную боль, а не ради коллекции;
- workflows описывают реально повторяемые процедуры;
- новые артефакты появляются после проверенных уроков.
