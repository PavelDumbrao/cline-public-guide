# 04. Skills Guide

## Что такое skill
Skill — это модуль знаний для отдельного типа задач.

Rule отвечает на вопрос **«как себя вести всегда»**.
Skill отвечает на вопрос **«как действовать в конкретной теме»**.

## Примеры
- `systematic-debugging` — как отлаживать
- `frontend-design` — как проектировать интерфейс
- `n8n-http-request` — как настраивать HTTP Request в n8n
- `supabase-integration` — как подключать Supabase

## Когда создавать новый skill
Создавай skill, если:
- ты повторяешь одну и ту же инструкцию в 3-й раз
- задача регулярно всплывает
- нужен единый шаблон действий

## Хороший skill содержит
- когда использовать
- когда не использовать
- ключевые правила
- типичные ошибки
- как проверить результат

## Типичная ошибка
Пихать skill в rules или писать туда личные ключи.

## Как проверить skill
Попроси Cline выполнить задачу, для которой skill предназначен, и проверь:
- действительно ли он использовал этот skill
- стал ли ответ более структурным
- появились ли практичные шаги вместо воды


## Как skill связан с rules
Rules отвечают за **always-on routing**:
- когда вообще нужен skill;
- когда skill не нужен;
- какие skills стоит предпочитать в разных классах задач.

Skill отвечает уже за **глубокую работу внутри темы**.

Простой способ думать так:
- Rule = «когда подключать экспертизу»
- Skill = «как выполнять задачу внутри этой экспертизы»

## Когда skill не надо создавать
Не создавай новый skill, если:
- ошибка была единичной и ещё не повторялась;
- достаточно короткого обновления в Rule;
- это должно решаться через Hook;
- это проектная заметка, incident log или handoff;
- уже есть существующий skill, который покрывает задачу.

## Как понять, что skill реально нужен
Skill оправдан, если одновременно выполняются 3 условия:
1. паттерн повторяется;
2. это не always-on policy;
3. без такого skill агент будет снова и снова изобретать тот же workflow.

## Как выглядит хороший skill практически
Хороший skill:
- короткий в ядре;
- содержит `when to use` и `when not to use`;
- не хранит секреты;
- не дублирует rules;
- может ссылаться на `docs/`, `templates/`, `scripts/` вместо простыни внутри `SKILL.md`.

## Эволюция skills
Система должна развиваться так:
- ошибка/паттерн обнаружен;
- ошибка исправлена;
- решение проверено;
- lesson candidate классифицирован;
- только потом обновляется existing skill или создаётся новый.

Если нужен отдельный meta-skill для этого процесса — используй `skill-authoring`.

## Мини-чеклист для skill review
- [ ] skill отвечает за одну понятную зону ответственности
- [ ] есть «когда использовать»
- [ ] есть «когда не использовать»
- [ ] нет секретов и личной инфраструктуры
- [ ] нет дублирования existing skills и rules
