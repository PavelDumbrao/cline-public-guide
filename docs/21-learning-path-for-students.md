## Шаг 1. Понять сущности
### Что изучить
- rules
- skills
- hooks
- MCP
- workflows

### Домашка
Объясни своими словами разницу между rule, skill и hook.

### Критерий понимания
Ты можешь без подсказки сказать, что куда относится.

### Типичный провал
Ученик сваливает всё в rules.

## Шаг 2. Поставить базовый setup
### Что сделать
- взять 5–7 rules
- взять 3–4 skills
- поставить 2–3 MCP

### Домашка
Собери минимальный setup без hooks.

### Критерий понимания
Ты не копируешь всё подряд, а понимаешь, зачем каждый элемент нужен.

### Типичный провал
Сразу поставить весь стек автора.

## Шаг 3. Научиться проверять setup
### Что сделать
Прогнать smoke tests на:
- language/style
- docs search
- skills
- MCP

### Домашка
Пройти готовые prompt-тесты из [`examples/prompts/smoke-tests.md`](../examples/prompts/smoke-tests.md).

### Критерий понимания
Ты умеешь доказать, что setup работает, а не просто «кажется, что работает».

### Типичный провал
Ничего не проверять после переноса.

## Шаг 4. Освоить MCP и browser
### Что сделать
Понять, когда нужен search, browser, docs, local code search.

### Домашка
Собрать свой минимальный MCP-набор под одну конкретную роль.

### Критерий понимания
Ты можешь объяснить, почему не ставишь лишний MCP.

### Типичный провал
Плодить мусорный MCP-стек.

## Шаг 5. Сделать первый мини-проект
### Что сделать
Собери маленький рабочий кейс с rules + skills + MCP.

### Домашка
Например: мини-репозиторий с README, одним skill и одним workflow.

### Критерий понимания
Ты можешь довести маленькую задачу до результата и проверить её.

### Типичный провал
Брать слишком большой проект первым кейсом.

## Шаг 6. Научиться делать self-check и review
### Что сделать
Отдельно проверять:
- результат
- риски
- утечки
- слабые места

### Домашка
После любой задачи оформлять короткий self-review.

### Критерий понимания
Ты не веришь агенту на слово.

### Типичный провал
Считать первый ответ уже финальным.

## Шаг 7. Научиться строить свои skills и workflows
### Что сделать
Понять, когда тема достойна отдельного skill или workflow.

### Домашка
Сделать один свой skill из повторяющейся задачи и один workflow из повторяющейся процедуры.

### Критерий понимания
Ты не просто копируешь шаблон, а умеешь адаптировать систему под себя.

### Типичный провал
Создавать десятки навыков без реальной необходимости.

## Шаг 8. Научиться эволюционировать систему, а не плодить хаос
### Что сделать
Понять, как система должна учиться на повторяющихся ошибках:
- когда добавлять новый rule;
- когда обновлять existing skill;
- когда нужен hook;
- когда достаточно project note.

### Домашка
Возьми один повторяющийся баг или рабочий паттерн и классифицируй его как:
- Rule
- Skill
- Hook
- Workflow
- note

### Критерий понимания
Ты не превращаешь каждый единичный случай в новый артефакт, а умеешь выбирать правильный слой знаний.

### Типичный провал
После каждой ошибки плодить новый skill, новый rule и новый hook одновременно.

## Шаг 9. Научиться видеть систему целиком
### Что сделать
Прочитать:
- [`docs/23-system-map-rules-skills-hooks-mcp.md`](23-system-map-rules-skills-hooks-mcp.md)

### Домашка
Нарисуй себе карту своей будущей системы:
- что у тебя будет Rules;
- какие Skills реально нужны;
- какие Hooks действительно оправданы;
- какие MCP ты оставишь минимально необходимыми.

### Критерий понимания
Ты можешь объяснить, как у тебя связаны layers системы, а не просто назвать список файлов.

### Типичный провал
Понимать отдельные части, но не понимать, как они работают вместе.

## Шаг 10. Научиться ставить setup без wrong-path ошибок
### Что сделать
Пройти три документа как единый слой:
- [`docs/25-cross-platform-installation-paths.md`](25-cross-platform-installation-paths.md)
- [`docs/26-mega-prompt-for-ai-installer.md`](26-mega-prompt-for-ai-installer.md)
- [`docs/27-setup-self-check-and-recovery.md`](27-setup-self-check-and-recovery.md)

### Домашка
Собери безопасный install-план для своего setup:
- где у тебя global rules;
- где workspace rules;
- где skills;
- где hooks;
- где MCP config;
- как ты будешь проверять visibility каждого слоя.

### Критерий понимания
Ты можешь объяснить не только «куда класть файлы», но и как доказать, что Cline реально видит каждый слой.

### Типичный провал
Файлы записаны, а ученик считает задачу закрытой, хотя visibility check даже не делался.

## Шаг 11. Научиться строить планы в 3 слоя
### Что сделать
Прочитать:
- [`28-three-level-planning-in-cline.md`](28-three-level-planning-in-cline.md)

Понять и уметь различать:
- **глобальный план** — master plan проекта;
- **средний план** — operational plan текущей фазы;
- **точечный чек-лист** — текущий исполнимый slice в `task_progress`.

### Домашка
Возьми одну реальную длинную задачу и оформи её в 3 слоя:
- `implementation_plan.md`
- `cline_docs/project-state.md`
- `task_progress`

### Критерий понимания
Ты можешь без путаницы объяснить:
- почему `task_progress` не должен быть master plan всего проекта;
- что именно хранится в `cline_docs/project-state.md`;
- когда глобальный план надо обновлять, а когда лучше не трогать;
- как из одной большой цели перейти к короткому исполнимому slice.

### Типичный провал
- ученик пишет весь проект в один TODO-список;
- превращает master plan в список из десятков микрошагов;
- не фиксирует definition of done для фаз;
- делает deep-planning в чате, но не переносит его в рабочие файлы проекта.
