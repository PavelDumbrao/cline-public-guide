# 19. Browser Automation Safety

## Зачем нужен browser automation
Browser automation полезен, когда нужно:
- проверить UI после изменений;
- увидеть ошибки в консоли;
- проверить network-запросы;
- прогнать базовый пользовательский сценарий.

## Когда Playwright реально полезен
- frontend / fullstack проекты;
- формы, кнопки, маршруты, кабинеты;
- отладка проблем, которые не видно по коду;
- smoke test после правок.

## Когда лучше не давать агенту браузерный доступ
- если речь про чувствительный аккаунт;
- если логин связан с финансами, личными данными, админкой продакшена;
- если нет ясной цели, зачем вообще нужен браузер;
- если задачу можно решить read-only способом.

## Read-only vs action-oriented сценарии

### Read-only
- открыть страницу;
- посмотреть DOM;
- проверить консоль;
- проверить network;
- сделать скриншот.

### Action-oriented
- кликнуть;
- заполнить форму;
- перейти по сценарию;
- отправить данные.

По умолчанию лучше мыслить browser automation как **read-only**, если нет явной цели на действие.

## Почему надо быть осторожным с cookies / сессиями / логинами
Потому что это уже не просто тест, а работа с доступом.
Здесь риск:
- случайных действий;
- утечки сессии;
- изменения данных;
- выполнения опасных операций не в том аккаунте.

## Почему нельзя бездумно давать доступ к чувствительным аккаунтам
Потому что даже хороший агент может:
- ошибиться в шаге;
- нажать не туда;
- неверно понять задачу;
- запустить действие там, где ожидался только просмотр.

## Как безопасно тестировать browser automation
1. Начни с read-only сценария
2. Проверяй консоль и network до кликов
3. Не логинься в чувствительные аккаунты без необходимости
4. Явно формулируй цель действия
5. После действия делай проверку результата

## Что проверять после действия агента в браузере
- что реально произошло;
- не отправлены ли лишние данные;
- не изменены ли критичные настройки;
- не ушёл ли агент в неверный сценарий.

## Что можно
- открыть страницу
- проверить консоль
- проверить network
- сделать скриншот
- пройти безопасный тестовый сценарий

## Что нельзя
- бездумно давать доступ к чувствительным аккаунтам
- считать, что browser automation безопасен по умолчанию
- выполнять чувствительные действия без явной команды пользователя

## Что требует явной команды пользователя
- логин в важный аккаунт
- действия с оплатой, доступами, удалением, публикацией
- любые необратимые действия
