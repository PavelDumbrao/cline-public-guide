## Что такое rule
Rule — это постоянная инструкция для AI.
Она говорит агенту, как вести себя и каких принципов придерживаться.

> Если хочешь проходить rules **по порядку, с базой, проверками и отчётом после каждого внедрения**, читай [`docs/24-rules-mega-guide.md`](24-rules-mega-guide.md).

## Зачем нужны rules
Rules уменьшают хаос.
Без них агент:
- чаще фантазирует
- сильнее расплывается
- может делать небезопасные вещи
- хуже держит стиль и формат работы

## Хорошие rules отвечают на вопросы
- на каком языке отвечать?
- как писать код?
- когда искать документацию?
- как тестировать?
- как обращаться с секретами?
- как использовать tools?

## Типичная ошибка
Складывать в rules всё подряд:
- инфраструктуру
- личные URL
- токены
- одноразовые incident notes

Так делать нельзя.

## Как понять, что rule хороший
- он короткий
- он конкретный
- его можно применять повторно
- в нём нет личных данных
- он помогает принимать решения

## Как проверить, что rule работает
Дай Cline простую задачу и посмотри:
- говорит ли он на нужном языке
- придерживается ли структуры
- ищет ли документацию когда нужно
- избегает ли секретов и опасных действий

## Always-on vs project-specific rules
Есть два полезных уровня rules:

### Always-on rules
Это правила, которые подходят почти везде:
- язык и стиль;
- базовая безопасность;
- поиск документации;
- работа с инструментами;
- базовое качество кода.

### Project-specific rules
Это правила под конкретный проект или инфраструктуру:
- текущий стек проекта;
- локальные ограничения;
- routing по внутренним сервисам;
- рабочие команды и локальные договорённости.

Важно: project-specific правило не должно превращаться в склад секретов и случайных заметок.

## Какие категории rules реально нужны
Обычно зрелая система правил покрывает:
- язык и стиль;
- поиск документации и проверяемость;
- качество кода;
- безопасность;
- работу с инструментами;
- управление контекстом;
- routing по skills;
- проектный workflow.

Это не значит, что новичку нужно сразу писать 20 файлов.
Это значит, что со временем rules обычно собираются именно вокруг этих зон.

## Как rule связан с остальными слоями

### Rule ↔ Skill
Rule говорит:
> когда вообще подключать skill

Skill говорит:
> как глубоко работать внутри темы

### Rule ↔ Hook
Rule задаёт policy.
Hook может автоматизировать её enforcement.

### Rule ↔ MCP
Rule может задавать порядок выбора инструментов:
- когда сначала читать локальный код;
- когда идти в docs;
- когда использовать веб-поиск;
- когда нужен browser.

### Rule ↔ Workflow
Workflow описывает повторяемую процедуру,
но сам опирается на уже существующие rules как на фоновые ограничения.

## Когда правило НЕ нужно создавать
Не создавай новый rule, если:
- это единичная ошибка;
- это узкая тема для одного skill;
- это лучше решается hook'ом;
- это локальная заметка по конкретному проекту;
- у тебя пока нет повторяемого паттерна.

## Как система правил должна эволюционировать
Нормальный путь такой:
1. ошибка или слабый паттерн найдены;
2. проблема исправлена;
3. исправление проверено;
4. причина понята;
5. только потом решается, нужен ли новый rule или достаточно обновить существующий.

Это защищает систему от хаотичного разрастания.

## Мини-чеклист review для rules
- [ ] правило решает повторяемую задачу, а не разовый случай
- [ ] в нём нет секретов и приватных значений
- [ ] оно помогает принимать решения, а не только «красиво звучит»
- [ ] оно не дублирует skill, hook или workflow
- [ ] его можно проверить через простой smoke test
