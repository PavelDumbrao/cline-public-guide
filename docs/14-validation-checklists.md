# 14. Validation Checklists

> Этот файл нужен, когда ты уже что-то добавил в систему и хочешь проверить не по ощущению, а по критериям: стало лучше или просто сложнее.

Сначала полезно прочитать:
- [`03-rules-guide-human.md`](03-rules-guide-human.md)
- [`07-mcp-guide-human.md`](07-mcp-guide-human.md)
- [`27-setup-self-check-and-recovery.md`](27-setup-self-check-and-recovery.md)

## Как правильно пользоваться этим файлом

Validation нужна не “для галочки”.

Она нужна, чтобы отличать:
- полезный новый слой;
- лишний слой;
- слой, который записан, но не работает;
- слой, который ухудшил предсказуемость системы.

Проверяй каждый компонент в трёх точках:
1. **до включения** — нужен ли он вообще;
2. **сразу после включения** — есть ли proof, что он работает;
3. **после пары реальных задач** — дал ли он пользу или только сложность.

---

## A. Проверка нового rule

### До включения
- rule reusable?
- rule достаточно конкретный?
- это точно rule, а не skill / note / incident log?
- в нём нет секретов?

### После включения
- always-on поведение стало яснее?
- rule реально применяется?
- нет конфликтов с другими rules?

### Красные флаги
- личные URL и токены;
- слишком длинный текст без необходимости;
- частный кейс замаскирован под общий закон.

### Когда откатывать
- поведение агента стало менее предсказуемым;
- появились конфликты между rules;
- rule мешает сильнее, чем помогает.

---

## B. Проверка нового skill

### До включения
- skill отвечает за отдельный класс задач?
- у него есть “когда использовать / когда не использовать”?
- есть ли типичные ошибки?
- он не дублирует существующий skill?

### После включения
- ответы по теме стали структурнее?
- skill реально вызывается по своей теме?
- нет ощущения, что он просто повторяет базовые rules?

### Красные флаги
- skill слишком общий;
- skill слишком личный;
- skill не даёт понятного сценария использования.

### Когда откатывать
- если skill только мешает;
- если он дублирует другой skill;
- если после него ответы стали менее стабильными.

---

## C. Проверка нового hook

### До включения
- hook действительно нужен?
- stdout возвращает только JSON?
- в hook нет захардкоженных секретов?
- hook быстрый и диагностируемый?

### После включения
- hook срабатывает там, где должен?
- нет ложных блокировок?
- нет дублирующего мусора в stdout?
- happy-path не сломан?

### Красные флаги
- hook печатает лишний текст в stdout;
- зависит от личной инфраструктуры;
- его трудно отключить и понять.

### Когда откатывать
- блокирует нормальную работу;
- создаёт шум;
- ломает базовый workflow.

---

## D. Проверка нового MCP

### До включения
- зачем он нужен?
- не дублирует ли он существующий MCP?
- требует ли токен / логин / чувствительный доступ?
- уровень доступа оправдан?

### После включения
- MCP реально вызывается?
- польза от него понятна?
- есть ли конкретный smoke test по назначению?
- не стало ли setup сложнее без практической выгоды?

### Красные флаги
- не можешь объяснить, зачем MCP нужен;
- MCP просит слишком много доступа;
- MCP дублирует уже существующий инструмент.

### Когда откатывать
- если MCP не используется;
- если он добавляет сложность без пользы;
- если он создаёт лишний security risk.

---

## E. Проверка install path / visibility

### До включения
- path определён осознанно?
- global vs workspace scope выбран правильно?
- есть план visibility check после записи?

### После включения
- файл прочитан обратно?
- слой реально влияет на runtime?
- skill / hook / MCP действительно видим, а не просто существует на диске?

### Красные флаги
- файл есть, но поведение не изменилось;
- config обновлён, а runtime его игнорирует;
- непонятно, какой runtime вообще читает этот path.

### Когда откатывать
- если уже непонятно, какой слой сломал setup;
- если visibility check не пройден, а поверх уже навесили новые слои.

## Общий чек-лист validation

- [ ] Я проверяю компонент до включения
- [ ] Я проверяю компонент сразу после включения
- [ ] Я умею назвать красные флаги
- [ ] Я понимаю, когда компонент нужно откатить
- [ ] Я не добавляю новые слои только потому, что “так круче”

---

## Самые частые ложные победы

### Ложная победа 1
Файл записан, значит всё работает.

### Ложная победа 2
Агент сказал “готово”, значит слой реально виден.

### Ложная победа 3
В конфиге появилась запись про MCP, значит MCP уже рабочий.

### Ложная победа 4
Папка skill существует, значит skill уже вызывается.

Во всех этих случаях нужна отдельная проверка видимости и поведения.

---

## Когда validation можно считать достаточной

Validation достаточно хорошая, если у тебя есть:
- read-back файла;
- runtime visibility check;
- хотя бы один proof через prompt / event / tool use;
- понимание, что именно изменилось в поведении системы.

Если этого нет — validation ещё не закончена.
