# 14. Validation Checklists

> Этот файл нужен, когда ты уже что-то добавил в систему и хочешь проверить не по ощущению, а по чек-листу: стало лучше или только сложнее.

Сначала полезно прочитать:
- [`03-rules-guide-human.md`](03-rules-guide-human.md) — если проверяешь rule-слой
- [`07-mcp-guide-human.md`](07-mcp-guide-human.md) — если проверяешь MCP

---

## A. Проверка нового rule

### Зачем нужен
Чтобы rule не превратился в мусорную, личную или опасную инструкцию.

### Что проверить до включения
- rule reusable?
- rule конкретный?
- в нём нет секретов?
- это действительно rule, а не skill/incident note?

### Что проверить после включения
- поведение AI стало понятнее?
- rule реально применяется?
- нет конфликта с другими rules?

### Красные флаги
- личные URL
- токены
- слишком длинный текст
- rule описывает частный кейс как общий закон

### Когда откатывать
- AI стал хуже понимать задачи
- появились конфликты между rules
- rule делает поведение менее предсказуемым

## B. Проверка нового skill

### Зачем нужен
Чтобы skill реально помогал, а не дублировал общие знания.

### Что проверить до включения
- skill отвечает за отдельный тип задач?
- есть ли когда использовать / когда не использовать?
- есть ли типичные ошибки?
- нет ли секретов?

### Что проверить после включения
- ответы стали структурнее?
- skill реально используется по теме?
- нет ли дублирования существующих skills?

### Красные флаги
- skill слишком общий
- skill слишком личный
- skill без сценариев использования

### Когда откатывать
- если skill только мешает
- если он дублирует другой skill
- если он ведёт к нестабильным ответам

## C. Проверка нового hook

### Зачем нужен
Чтобы hook не ломал workflow и не давал ложных срабатываний.

### Что проверить до включения
- hook действительно нужен?
- stdout возвращает только JSON?
- нет ли захардкоженных секретов?
- hook быстрый и понятный?

### Что проверить после включения
- hook срабатывает там, где должен
- нет ложных блокировок
- нет дублирующего мусора в выводе

### Красные флаги
- hook печатает лишний текст в stdout
- hook зависит от личной инфраструктуры
- hook трудно отключить или понять

### Когда откатывать
- блокирует нормальную работу
- создаёт шум и дубль-сообщения
- ломает базовый workflow

## D. Проверка нового MCP

### Зачем нужен
Чтобы не тащить лишние и опасные интеграции.

### Что проверить до включения
- зачем нужен MCP?
- не дублирует ли он другой MCP?
- требует ли токены/аккаунт?
- безопасен ли он по объёму доступа?

### Что проверить после включения
- MCP реально вызывается
- польза от него понятна
- нет лишнего шума в setup

### Красные флаги
- не можешь объяснить зачем MCP нужен
- MCP требует слишком много доступа
- MCP дублирует уже существующий

### Когда откатывать
- если MCP не используется
- если он добавляет сложность без пользы
- если вызывает проблемы с безопасностью


## E. Проверка install path / visibility

### Зачем нужен
Чтобы отличать кейс «файл записан» от кейса «Cline реально видит слой».

### Что проверить до включения
- path определён осознанно?
- scope выбран правильно: global или workspace?
- есть план visibility check после записи?

### Что проверить после включения
- файл прочитан обратно?
- слой реально влияет на поведение?
- MCP / skill / hook реально вызывается, а не просто существует на диске?

### Красные флаги
- слой записан, но поведение не изменилось
- в config есть запись, но runtime её игнорирует
- непонятно, какой runtime вообще читает этот path

### Когда откатывать
- если уже непонятно, какой именно слой сломал setup
- если visibility check не пройден, а поверх уже добавлены новые слои
