# 02. Official vs Pavel Setup

## Зачем этот файл
Этот репозиторий основан на реальном рабочем setup, но важно честно отделить:
- официальный подход Cline
- рабочую практику автора
- то, что стоит брать ученикам
- то, что является личной кастомизацией

## Что ближе к официальной логике Cline
- rules как markdown-файлы
- skills как папки с `SKILL.md`
- hooks как bash-скрипты на события
- MCP как отдельные подключаемые серверы
- раздельная работа человека и AI через инструкции

## Что является практикой автора, а не стандартом
- большая пользовательская система правил
- своя методология под русскоязычный workflow
- интеграции с конкретной инфраструктурой
- собственные паттерны Telegram / VPS / n8n / AI-to-AI взаимодействия
- хранение большого количества рабочих артефактов вне «идеально чистой» структуры

## Где были отличия в исходном setup
- global rules хранились в отдельной пользовательской директории
- workspace rules хранились в `.clinerules/`
- workflow JSON для n8n лежали россыпью на Desktop
- часть rules/skills была завязана на личную инфраструктуру

## Таблица различий

| Тема | Официальная логика Cline | Практика Павла | Что брать ученику |
|---|---|---|---|
| Rules | Markdown-инструкции для поведения AI | Большой модульный набор правил | Брать модульные reusable rules без личной инфры |
| Skills | Папки с `SKILL.md` | Большой практический skill-stack под реальные кейсы | Брать только общие и понятные skills |
| Hooks | Событийные скрипты | Hooks для логирования, защиты и уведомлений | Начинать с минимальных безопасных hooks |
| Workflows | Повторяемые markdown-процессы для Cline | В рабочей системе акцент был больше на rules/skills, а не на публичных workflow-шаблонах | Брать markdown workflows как reusable procedures |
| MCP | Подключаемые серверы/инструменты | Широкий набор под docs, search, browser, GitHub, n8n, infra | Начинать с must-have MCP, не ставить всё подряд |
| Хранение файлов | Логичная структура по типам сущностей | Часть артефактов жила в кастомных директориях и на Desktop | Держать чистую структуру repo, не копировать хаос |
| Telegram / n8n / browser automation | Опциональные сценарии | Активно использовались в живой практике | Брать только как шаблоны, без личных токенов и URL |
| Headless mode | Продвинутый сценарий | Завязан на конкретные процессы автора | Не брать на старте, только после базового setup |
| Personal tech-stack | Не является стандартом Cline | Есть сильная личная инфраструктурная специфика | Не копировать буквально |

## Что ученикам лучше брать как стандарт
1. Держать публичный repo чистым и шаблонным
2. Разделять «для человека» и «для AI»
3. Не класть секреты в rules / skills / hooks
4. Не путать Cline Workflows и n8n JSON
5. Начинать с минимального MCP-стека

## Что не нужно копировать буквально
- личные пути
- VPS-процессы автора
- личный Telegram-бот
- private-only MCP
- incident notes и технические следы живой эксплуатации

## Рекомендуемый стандарт для ученика

### Чеклист
- [ ] взять минимальный набор rules
- [ ] взять 3–4 skills под свои задачи
- [ ] поставить 2–3 MCP, а не десять
- [ ] hooks подключать только после базовой проверки
- [ ] не копировать чужой tech-stack 1-в-1
- [ ] все приватные данные заменить вручную

## Рекомендация
Используй этот репозиторий как **каркас**, а не как готовый перенос чужой жизни в свой проект.
