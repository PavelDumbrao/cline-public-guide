# 01. Cline Mental Model

## Роль Cline
Cline удобно мыслить как **исполнителя**.

Не как всезнающего архитектора.
Не как абсолютный источник истины.
А как сильный AI-инструмент, который хорошо:
- исполняет задачу
- читает проект
- меняет файлы
- проверяет гипотезы
- работает по инструкциям

## Практичная связка

### GPT / Perplexity / другой LLM
Используй как:
- архитектора
- генератор вариантов
- критика
- исследователя

### Cline
Используй как:
- исполнителя
- редактора проекта
- оператора CLI
- проверяющего локальную систему

## Рабочий цикл
1. Сначала формулируешь задачу
2. Потом просишь LLM предложить архитектуру
3. Потом даёшь Cline конкретную подзадачу
4. Потом отдельно проверяешь результат

## Критически важно
**Не верь агенту на слово.**

Проверяй:
- файл действительно создан?
- команда реально отработала?
- тесты реально зелёные?
- страница реально открывается?
- workflow реально импортируется?

## Чем отличаются сущности

### Rule
Глобальное правило поведения.
Пример: «всегда отвечай по-русски».

### Skill
Узкоспециализированная инструкция.
Пример: «как работать с n8n HTTP Request node».

### Hook
Автоматический bash-обработчик события.
Пример: «перед выполнением команды проверить, нет ли `git add .env`».

### MCP
Подключаемый инструмент/сервер данных.
Пример: документация библиотек, GitHub, браузер, веб-поиск.

### Cline Workflow
Пошаговый markdown-сценарий для самого Cline.

### n8n Workflow JSON
Импортируемый workflow для n8n.

## Главная ошибка новичка
Путать эти уровни и пытаться засунуть всё в rules.
