> Это один из самых важных файлов для ученика. Он отвечает на вопрос: **какие внешние инструменты вообще стоит подключать к Cline, а какие пока не трогать**.

## Зачем этот файл нужен

Новичок часто делает одну из двух ошибок:
- либо вообще не понимает, что такое MCP
- либо подключает слишком много MCP сразу и потом сам не понимает, что из этого реально полезно

Этот файл нужен, чтобы выбрать **минимальный полезный стек**, а не собирать коллекцию инструментов ради галочки.

## Что такое MCP
MCP — это способ подключить к Cline внешние инструменты и источники данных.

## Важный факт: у Cline есть MCP Marketplace
У Cline есть **MCP Marketplace** — каталог готовых MCP-серверов, которые можно найти, установить и настроить намного проще, чем вручную собирать всё по чужим репозиториям.

Официальные ссылки:
- [Cline MCP Marketplace](https://docs.cline.bot/mcp/mcp-marketplace)
- [Adding & Configuring Servers](https://docs.cline.bot/mcp/adding-mcp-servers-from-github)

Что это значит на практике:
- ты можешь сначала искать MCP прямо через Marketplace;
- часто подключение уже подготовлено в удобном формате;
- во многих случаях достаточно вставить **API key** или пройти **логин / OAuth**;
- Cline обычно подсказывает, **какая переменная окружения нужна** и где смотреть setup-инструкцию.

Простое правило для новичка:
если тебе нужен популярный MCP вроде `tavily`, `exa`, `github` или `playwright` — **сначала проверь Marketplace**, а не начинай с ручного поиска по всему интернету.

Полезные формулировки для Cline:
- «Покажи, как подключить Tavily MCP через Marketplace»
- «Покажи официальный setup для Exa MCP и где взять API key»
- «Покажи, нужен ли для GitHub MCP token или достаточно логина в моём случае»

## Чем MCP отличается от skills / hooks / workflows
- MCP даёт инструмент или данные
- skill даёт знания и шаблон мышления
- hook автоматически запускает скрипт на событии
- workflow задаёт шаги работы

## Если ты новичок: что такое GitHub и что такое репозиторий
GitHub — это платформа, где хранят и развивают код проектов.

Если очень просто:
- **GitHub** — это место, где лежат проекты, их история, ветки, issues и pull requests;
- **репозиторий** — это конкретный проект или папка проекта с файлами, кодом и историей изменений.

Примеры:
- репозиторий сайта;
- репозиторий Telegram-бота;
- репозиторий n8n-автоматизации;
- репозиторий публичного гайда, как этот.

Зачем это знать в контексте MCP:
- `github` MCP нужен не «для интернета вообще», а именно для работы с **репозиториями на GitHub**;
- через него Cline может читать issues, pull requests, коммиты, файлы и иногда вносить изменения, если у тебя есть доступ;
- если ты вообще не работаешь с GitHub-репозиториями, этот MCP тебе может быть не нужен на старте.

## Must-have MCP для новичка
- `context7` — документация библиотек
- `tavily` — веб-поиск и research
- `playwright` — браузер и UI-проверки

### Почему именно они
- `context7` закрывает проблему устаревших догадок по библиотекам
- `tavily` помогает искать актуальную информацию в вебе
- `playwright` помогает не гадать по UI, а реально проверять страницу

## Optional MCP
- `github` — если работаешь с GitHub
- `exa` — если нужен чистый поиск
- `cocoindex-code` — если у тебя большая локальная кодовая база
- `n8n-docs` — если работаешь с n8n

## Advanced MCP
- `n8n-mcp` — для live-инстанса n8n
- второй Exa MCP, если ты понимаешь зачем он нужен

## Private-only MCP
- `hostinger-api`

## Типовые сценарии выбора MCP

### Мне нужен только docs + web search
**Бери:** `context7`, `tavily`
**Не ставь:** `playwright`, `n8n-mcp`, `hostinger-api`

### Я делаю frontend и хочу browser automation
**Бери:** `context7`, `tavily`, `playwright`, опционально `github`
**Не ставь:** `n8n-mcp`, `hostinger-api`

### Я сижу в n8n
**Бери:** `context7`, `tavily`, `n8n-docs`, при необходимости `n8n-mcp`
**Не ставь:** `hostinger-api`, лишние search MCP-дубликаты

### У меня большой локальный repo
**Бери:** `context7`, `tavily`, `cocoindex-code`, опционально `github`
**Не ставь:** `n8n-mcp`, если не работаешь с n8n

### Мне не нужен self-hosted / infra layer
**Бери:** стандартный набор без infra-specific MCP
**Не ставь:** `hostinger-api`, любые private-only интеграции

## Какие требуют API keys или логин
Чаще всего требуют:
- tavily
- exa
- github
- n8n live access
- hostinger-api

Но важен нюанс:
- **не каждый MCP требует ключ**;
- некоторые MCP локальные и работают без внешнего API;
- у некоторых ключ нужен не для старта, а для higher limits или production-сценария.

Safest path такой:
1. Сначала открой Marketplace.
2. Посмотри официальный install flow.
3. Только потом заводи ключ или логин.
4. Сразу после подключения сделай маленький smoke test.

## Как безопасно добавлять MCP
1. Подключай по одному
2. Понимай, зачем он нужен
3. Проверяй, не дублирует ли он уже существующий MCP
4. Не храни секреты в repo
5. Проверяй, какие данные MCP может читать и отправлять

## В каком порядке обычно подключать MCP
Практически почти всегда работает такой порядок:

1. **Context7** — если нужен код, библиотека, API, SDK, фреймворк.
2. **Tavily** — если нужен актуальный веб-поиск и general research.
3. **Playwright** — если нужно реально проверить UI, форму, консоль, сеть.
4. **GitHub** — если работаешь с репозиториями, PR, issues, code search.
5. **Exa** — если нужен альтернативный чистый search/code context.
6. **Infra-specific MCP** — только когда уже реально есть такой use-case.

Идея простая:
сначала подключай то, что закрывает самую частую боль,
а не то, что звучит «продвинуто».

## Как думать про ключи без хаоса

Не надо заранее собирать все ключи подряд.

Нормальный путь такой:
1. Сначала понять, нужен ли тебе этот MCP вообще.
2. Потом открыть его официальный install flow.
3. Только потом завести ключ или логин.
4. Сразу проверить MCP маленьким smoke test.

Пример здравого подхода:
- нужен web search → сначала ставишь `tavily`;
- нужен чистый альтернативный search → потом думаешь про `exa`;
- работаешь с GitHub repo и PR → тогда уже подключаешь `github`;
- есть live n8n instance → только тогда идёшь в `n8n-mcp`.

## Как выбирать между похожими MCP

### Context7 vs Tavily
- **Context7** — для документации библиотек и framework API.
- **Tavily** — для широкого внешнего поиска и свежей информации.

### Tavily vs Exa
- **Tavily** — основной поиск по умолчанию.
- **Exa** — дополнительный чистый search/code context, если Tavily уже недостаточно.

### GitHub vs локальный code search
- **GitHub MCP** — когда нужен remote repo, PR, issues, public code search.
- **cocoindex-code** — когда нужен semantic search по локальной кодовой базе.

### Playwright vs search/extract
- **Playwright** — когда нужен интерактив, клики, формы, консоль, сеть.
- **Search/extract** — когда страницу нужно просто прочитать, а не прожить как пользователь.

## Типовые задачи → какой MCP брать

| Задача | Что брать первым |
|---|---|
| Документация библиотеки | `context7` |
| Актуальная информация из веба | `tavily` |
| Чистый технический search / code context | `exa` |
| UI-проверка и браузерный сценарий | `playwright` |
| Работа с GitHub repo / PR / issues | `github` |
| Семантический поиск по большому локальному repo | `cocoindex-code` |
| n8n documentation / validation | `n8n-docs` |
| Live-операции с n8n | `n8n-mcp` |

## Что особенно важно про Marketplace

Marketplace полезен не только как каталог, а как **быстрый фильтр от лишних интеграций**.

Если MCP есть в Marketplace, это ещё не значит, что он тебе нужен.

Сначала спрашивай себя:
- это must-have или просто интересно посмотреть?
- он решает мою реальную задачу или дублирует уже установленный MCP?
- это безопасный read-only MCP или он даёт доступ к живым сервисам?

Хороший сценарий для новичка:
1. Открыть Marketplace.
2. Выбрать 2–3 MCP максимум.
3. Подключить их по одному.
4. Прогнать smoke tests.
5. Только потом расширять стек.

## Как не утонуть в MCP при первом setup
Новичку почти никогда не нужен огромный стек.

Нормальный путь:
1. Возьми 2–3 MCP.
2. Проверь их простыми smoke test-сценариями.
3. Поработай с ними на реальных задачах.
4. Только потом решай, чего реально не хватает.

Если после подключения нового MCP твоя система не стала заметно полезнее — скорее всего, он пока лишний.

## Минимальные smoke tests после подключения MCP

### Context7
Попроси найти актуальную документацию по известной библиотеке.

### Tavily
Попроси найти свежую информацию по конкретной технической теме.

### Playwright
Попроси открыть страницу и проверить видимый результат действия.

### GitHub
Попроси прочитать issue, PR или найти код по репозиторию.

### n8n-docs
Попроси найти нужную ноду и объяснить её обязательные поля.

### cocoindex-code
Попроси найти реализацию по смыслу, а не по точному имени.

## Практическое правило безопасности
Чем мощнее MCP, тем внимательнее нужно понимать его границы доступа.

Особенно осторожно относись к MCP, которые умеют:
- писать в GitHub;
- деплоить;
- работать с хостингом;
- менять live-сервисы;
- читать приватные данные.

Такие MCP не надо подключать «на всякий случай».

- Не ставь MCP «на всякий случай»
