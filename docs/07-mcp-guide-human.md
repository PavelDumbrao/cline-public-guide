# 07. MCP Guide

## Что такое MCP
MCP — это способ подключить к Cline внешние инструменты и источники данных.

## Чем MCP отличается от skills / hooks / workflows
- MCP даёт инструмент или данные
- skill даёт знания и шаблон мышления
- hook автоматически запускает скрипт на событии
- workflow задаёт шаги работы

## Таблица MCP

| MCP | Зачем нужен | Кому нужен | Нужен API key? | Когда не ставить |
|---|---|---|---|---|
| context7 | Актуальная документация библиотек | Почти всем | Обычно нет | Если вообще не работаешь с кодом/библиотеками |
| tavily | Веб-поиск, research, извлечение контента | Почти всем | Да | Если не нужен веб-поиск |
| playwright | Проверка UI, браузер, консоль, сеть | Frontend/fullstack/browser automation | Нет | Если не работаешь с UI |
| github | Репозитории, PR, issues, code search | Тем, кто работает с GitHub | Да/логин | Если не используешь GitHub |
| exa | Чистый веб-поиск и code context | Тем, кому нужен альтернативный поиск | Да | Если уже хватает Tavily |
| cocoindex-code | Семантический поиск по локальному коду | Тем, у кого большие кодовые базы | Нет/не всегда | Если проекты маленькие |
| n8n-docs | Документация и валидация n8n нод | Пользователям n8n | Обычно да | Если ты не работаешь с n8n |
| n8n-mcp | Работа с живым n8n-инстансом | Продвинутым n8n-пользователям | Да | Если нет live n8n use-case |
| hostinger-api | Управление Hostinger через API | Только infra-specific пользователям | Да | Почти всегда не ставить новичку |

## Must-have MCP для новичка
- `context7` — документация библиотек
- `tavily` — веб-поиск и research
- `playwright` — браузер и UI-проверки

## Optional MCP
- `github` — если работаешь с GitHub
- `exa` — если нужен чистый поиск
- `cocoindex-code` — если у тебя большая локальная кодовая база
- `n8n-docs` — если работаешь с n8n

## Advanced MCP
- `n8n-mcp` — для live-инстанса n8n
- второй Exa MCP, если ты понимаешь зачем он нужен

## Private-only MCP
- `hostinger-api`

## Как безопасно добавлять MCP
1. Подключай по одному
2. Понимай, зачем он нужен
3. Проверяй, не дублирует ли он уже существующий MCP
4. Не храни секреты в repo
5. Проверяй, какие данные MCP может читать и отправлять

## Как не плодить мусорный MCP-стек
- Не ставь MCP «на всякий случай»
- Не держи два одинаковых MCP без понятной причины
- Если не можешь объяснить, зачем нужен MCP — пока не ставь
- Сначала проверь, закрывает ли твою задачу уже установленный MCP
- Начинай с минимума и добавляй только под конкретную боль

## Типичная ошибка
Ставить 10 MCP сразу и потом не понимать, какой из них реально нужен.
