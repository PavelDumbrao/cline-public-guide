# README FOR CLINE

> Это инструкция для AI-агента, который читает этот публичный репозиторий как шаблон настройки Cline.

Ты работаешь внутри публичного учебного репозитория по Cline.

## Цель
Помогать пользователю настраивать и адаптировать Cline безопасно, без утечки секретов и без слепого копирования чужой инфраструктуры.

## Что важно
- Всегда различай rules, skills, hooks, MCP и workflows.
- Не путай Cline Workflows и n8n workflow JSON.
- Не предлагай копировать личный tech stack автора как обязательный стандарт.
- Не храни секреты в markdown, skill-файлах, rules и hooks.
- Если встречаешь placeholders — ожидай, что пользователь должен заменить их вручную.

- Перед установкой любого слоя сначала смотри `../docs/25-cross-platform-installation-paths.md`.
- Перед тем как собирать setup за пользователя, опирайся на `../docs/26-mega-prompt-for-ai-installer.md`.
- Если слой записан, но не видно, переходи к `../docs/27-setup-self-check-and-recovery.md`, а не объявляй работу завершённой.

## Как использовать этот репозиторий
1. Сначала читай `ai/rules/`.
2. Потом при необходимости подключай `ai/skills/`.
3. Если задача повторяющаяся — используй `ai/workflows/`.
4. Если нужна автоматизация событий — смотри `ai/hooks/`.
5. Для интеграции с n8n используй примеры из `examples/n8n/` только как шаблоны.

## Как должен работать routing между слоями
- **Rules** определяют базовую политику и говорят, когда нужен skill.
- **Skills** дают глубокую on-demand экспертизу внутри темы.
- **Hooks** используются, если нужна детерминированная автоматизация и enforcement.
- **Workflows** используются для повторяемых многошаговых процедур.
- **MCP** подключаются как внешние инструменты, а не как замена rules/skills.

Если паттерн повторяется, сначала оцени: это Rule, Skill, Hook, Workflow или project note?
Не создавай новый артефакт хаотично.

## Если пользователь хочет собрать свой setup по этому repo
- Сначала сделай аудит его задач и текущего проекта.
- Потом предложи **минимальный** набор rules / skills / MCP.
- Не копируй весь репозиторий механически.
- Всё адаптируй под задачи пользователя и его стек.
- После каждого этапа проверяй результат, а не просто объявляй «готово».
- Если нужны пользовательские значения — оставляй placeholders и явно говори, что заменить вручную.

## Предпочтительный порядок адаптации
Если пользователь хочет перенести систему из этого repo, действуй так:
1. Определи его задачи, стек и текущий setup.
2. Выдели минимальный набор **rules**.
3. Добавь только нужные **skills**.
4. Подключи минимальные **MCP** под реальные use-cases.
5. Добавляй **hooks** только если есть понятная причина и проверка.
6. Используй **workflows** для повторяемых процедур, а не как замену rules/skills.
7. Для установки слоёв используй связку `docs/25` → `docs/26` → `docs/27`, а не разрозненные догадки по путям и runtime.

Не пытайся переносить всё за один проход.
Система должна становиться понятнее и устойчивее, а не просто больше.

## Что считать хорошим результатом
Хороший результат — это не «в репозитории стало много файлов».
Хороший результат — это когда:
- пользователь понимает, что у него включено всегда;
- какие штуки подключаются по ситуации;
- что автоматизировано хуками;
- какие MCP реально нужны;
- как он проверит, что setup действительно работает.

## Главные принципы
- Практичность важнее красоты.
- Проверка результата обязательна.
- Минимальная рабочая конфигурация лучше перегруженной.
- Всё личное должно быть заменено шаблонами.
- Если найден проверенный и повторяемый урок, система должна эволюционировать осознанно: через правильный слой знаний, а не через хаотичное накопление файлов.
