# README FOR CLINE

> Это инструкция для AI-агента, который читает этот публичный репозиторий как шаблон настройки Cline.

Ты работаешь внутри публичного учебного репозитория по Cline.

## Главная цель
Помогать пользователю собирать и адаптировать Cline безопасно:
- без утечки секретов;
- без слепого копирования чужой инфраструктуры;
- без хаотичного переноса всех слоёв сразу;
- с обязательной проверкой runtime visibility.

---

## Два основных режима

### 1. Human learning mode
Пользователь изучает repo как человек:
- читает `docs/`;
- понимает layers системы;
- сам решает, что переносить в свой setup.

### 2. AI sync mode
Пользователь даёт тебе repo как source of truth для переноса setup.

Тогда ты работаешь так:
1. изучаешь human docs как карту;
2. используешь `ai/` как syncable public layer;
3. переносишь setup по слоям;
4. после каждого слоя делаешь read-back и visibility check.

Для этого режима смотри:
- `../docs/30-ai-agent-sync-pack.md`

---

## Что ты обязан различать

Всегда различай:
- **Rules** — always-on policy;
- **Skills** — on-demand экспертиза;
- **Hooks** — event-driven automation / enforcement;
- **Workflows** — повторяемые markdown-процессы;
- **MCP** — внешние инструменты.

Не путай:
- Cline Workflows и n8n workflow JSON;
- “файл записан” и “runtime реально видит слой”.

---

## Как использовать этот repo правильно

### Сначала читай карту
Перед любым переносом прочитай:
- `README.md`
- `../docs/23-system-map-rules-skills-hooks-mcp.md`
- `../docs/13-install-and-adapt.md`
- `../docs/25-cross-platform-installation-paths.md`
- `../docs/26-mega-prompt-for-ai-installer.md`
- `../docs/27-setup-self-check-and-recovery.md`

### Потом определи минимум
Перед записью файлов ответь:
- что пользователю реально нужно сейчас;
- какие слои можно не трогать;
- какие project-specific values он должен заменить вручную;
- что должно быть global, а что workspace-specific.

### Потом устанавливай только слоями
Предпочтительный порядок:
1. rules
2. skills
3. MCP
4. hooks
5. workflows

Не пытайся переносить всё одним проходом.

---

## Что считать обязательным proof после каждого слоя

После каждого этапа ты должен уметь показать:
- `file_written: yes/no`
- `file_read_back: yes/no`
- `runtime_visibility_confirmed: yes/no`
- `proof_used: ...`
- `unresolved_risks: ...`

Если runtime visibility не подтверждена — слой не считается установленным.

---

## Что особенно важно

- Не копируй личный tech stack автора как обязательный стандарт.
- Не храни секреты в markdown, rules, skills и hooks.
- Если видишь placeholders — ожидай, что пользователь заменит их вручную.
- Не расширяй setup поверх неподтверждённого слоя.
- Если путь, runtime или visibility неясны — остановись и уточни.

Перед установкой любого слоя опирайся на:
- `../docs/25-cross-platform-installation-paths.md`
- `../docs/26-mega-prompt-for-ai-installer.md`
- `../docs/27-setup-self-check-and-recovery.md`

## Как должен работать routing между слоями

- **Rules** определяют базовую политику и routing.
- **Skills** дают глубокую on-demand экспертизу.
- **Hooks** используются, если нужна детерминированная автоматика и enforcement.
- **Workflows** применяются для повторяемых многошаговых процедур.
- **MCP** подключаются как внешние capabilities, а не как замена rules и skills.

Если повторяется паттерн, сначала оцени: это Rule, Skill, Hook, Workflow или project note?
Не создавай новый артефакт хаотично.

---

## Если пользователь хочет собрать свой setup по этому repo

Твоя безопасная последовательность такая:
1. Сделай аудит задач, стека и текущего setup.
2. Предложи минимальный набор rules.
3. Добавь только нужные skills.
4. Подключи минимальные MCP под реальные use-cases.
5. Добавляй hooks только если есть явная причина и проверка.
6. Используй workflows для повторяемых процедур, а не как замену rules / skills.
7. Если пользователь хочет public mirror setup, используй `ai/rules/`, `ai/skills/`, `ai/hooks/`, `ai/mcp/` как sync-pack layer.

Не пытайся переносить всё за один проход.

---

## Что нельзя делать

Не делай вот так:
- не объявляй “готово”, если пользователь не видит слой в интерфейсе;
- не маскируй низкую уверенность под уверенный тон;
- не копируй private-only values;
- не тащи hooks раньше, чем подтверждён базовый runtime;
- не пиши один и тот же слой сразу в несколько candidate paths без recovery-логики.

---

## Что считать хорошим результатом

Хороший результат — это не “создано много файлов”.

Хороший результат — это когда:
- пользователь понимает, что у него работает always-on;
- знает, какие штуки подключаются по ситуации;
- понимает, что автоматизировано хуками;
- знает, какие MCP реально нужны;
- видит proof, что setup действительно работает.

---

## Главные принципы

- Практичность важнее объёма.
- Проверка результата обязательна.
- Минимальная рабочая конфигурация лучше перегруженной.
- Всё личное должно быть заменено шаблонами.
- Система должна эволюционировать через проверенные уроки и правильный слой фиксации.
