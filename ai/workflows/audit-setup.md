# /audit-setup

## Цель
Провести аудит текущего setup Cline и получить практичный снимок состояния системы.

## Когда использовать
- Перед публикацией setup
- Перед большим рефакторингом rules/skills/hooks
- Когда нужно понять, что реально установлено и используется

## Входные данные
- доступ к текущему repo / директории setup
- список rules, skills, hooks, MCP
- при наличии: workflow examples и docs

## Пошаговый процесс
1. Собрать структуру директорий
2. Проверить rules: что активно, что личное, что reusable
3. Проверить skills: что полезно, что слишком project-specific
4. Проверить hooks: события, поведение, риски
5. Проверить MCP: must-have / optional / advanced / private-only
6. Отдельно найти sensitive info
7. Развести Cline workflows и n8n JSON
8. Сформировать итоговый отчёт

## Что считать хорошим результатом
- понятно, что входит в setup
- понятно, что можно публиковать
- понятно, что надо обезличить
- нет путаницы между сущностями

## Что проверить после выполнения
- не пропущены ли private-only элементы
- есть ли разделение для человека и для AI
- понятен ли итог новичку

## Типичные ошибки
- смешать n8n JSON и Cline workflows
- не заметить личную инфраструктуру в markdown
- считать всё установленное одинаково полезным

## Ограничения по безопасности
- не печатай живые секреты
- не публикуй токены, IP, домены, webhook URLs
- всё sensitive замещай placeholders
