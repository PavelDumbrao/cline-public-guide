# /review-rules

## Цель
Проверить набор rules на полезность, чистоту и отсутствие приватных данных.

## Когда использовать
- перед публикацией rules
- когда rules стали слишком большими
- когда агент ведёт себя нестабильно или противоречиво

## Входные данные
- список rule-файлов
- текущий контекст использования
- понимание, какие правила generic, а какие project-specific

## Пошаговый процесс
1. Собери все rules
2. Для каждого rule зафиксируй цель
3. Проверь, reusable ли он
4. Отметь личные/приватные привязки
5. Найди повторы и конфликты
6. Убери incident notes и частные кейсы из общих rules
7. Проверь, не пытается ли rule быть skill'ом
8. Подготовь сокращённый и понятный набор

## Что считать хорошим результатом
- rules короткие и конкретные
- rules не конфликтуют между собой
- rules не содержат секретов
- правила реально помогают принимать решения

## Что проверить после выполнения
- стало ли проще объяснить новичку, зачем каждый rule нужен
- нет ли расплывчатых и бессмысленных правил

## Типичные ошибки
- складывать в rules всё подряд
- хранить личный tech-stack внутри общего rule-набора
- описывать одноразовую проблему как глобальное правило

## Ограничения по безопасности
- не публикуй личные данные
- не оставляй секреты в markdown
- private-only части выноси отдельно или убирай
