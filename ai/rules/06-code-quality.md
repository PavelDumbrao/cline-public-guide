# 🛡️ Качество кода

> Это правило нужно, чтобы агент не оставлял после себя хаос: дубли, хрупкие правки, неявные ошибки и случайно сломанные части проекта.

Сначала полезно прочитать:
- [`01-core-language-style.md`](01-core-language-style.md) — базовый стиль и поведение
- [`07-tool-usage.md`](07-tool-usage.md) — как правильно использовать инструменты до и после изменений
- [`04-web-development.md`](04-web-development.md) — если изменения касаются интерфейса

---

## Зачем это правило нужно

Новичок часто радуется, когда «оно заработало».
Но хороший результат — это не только рабочий код, а ещё и:
- понятный код
- безопасные правки
- отсутствие лишнего дублирования
- проверка результата после изменений

---

## 🧱 Базовые правила
- Сначала изучай существующий код.
- Не дублируй уже существующую логику.
- Пиши минимальные и безопасные изменения.

### Что это значит на практике
- сначала найти, как похожая задача уже решалась в проекте
- не создавать второй `formatDate()`, если первый уже есть
- не переписывать полсистемы, если можно исправить один локальный участок

---

## ⚠️ Ошибки и внешние вызовы
- Обрабатывай ошибки явно.
- Давай понятный контекст ошибки.
- Не хардкодь секреты.

### Мини-примеры
Неправильно:
- молча проглотить ошибку
- вернуть просто `Something went wrong`
- вставить API key прямо в код

Правильно:
- сообщить, где именно произошла ошибка
- добавить контекст: какой запрос, какой файл, какой шаг
- использовать переменные окружения

---

## ✅ Проверка
- После изменений проверь, что результат реально работает.
- Если есть тесты — запускай их.
- Если менял поведение интерфейса — проверь браузер.
- Если менял backend-логику — проверь зависимые места.

---

## ✅ Чек-лист для самопроверки
- [ ] Я изучил существующую реализацию до правок
- [ ] Я не продублировал уже существующую логику
- [ ] Ошибки обрабатываются явно и с контекстом
- [ ] В коде нет хардкоженных секретов
- [ ] После правок я реально проверил результат
