# 🔎 Search Tools

> Это правило помогает не просто «искать что-то в интернете», а выбирать **правильный инструмент под правильную задачу**.

Сначала полезно прочитать:
- [`docs/00-start-here.md`](../../docs/00-start-here.md) — базовый маршрут для новичка
- [`docs/07-mcp-guide-human.md`](../../docs/07-mcp-guide-human.md) — человеческое объяснение, что такое MCP и как выбирать инструменты

---

## Зачем это правило нужно

Новичок часто делает одну из двух ошибок:
- либо вообще не ищет и начинает придумывать ответ из головы
- либо использует слишком тяжёлый инструмент там, где хватило бы простого

Это правило нужно, чтобы агент:
- искал быстрее
- искал точнее
- не тратил лишний контекст и время

---

## 🧭 Принцип выбора инструмента
- **Документация библиотеки** → Context7
- **Веб-поиск и research** → Tavily / Exa
- **Поиск по локальной кодовой базе** → semantic/local code search
- **Проверка UI и поведения страницы** → Playwright

---

## Когда какой инструмент брать

### Context7
Используй, когда нужен актуальный ответ по библиотеке, API или фреймворку.

Пример:
- «Как правильно настроить Next.js middleware?»
- «Какой синтаксис у Supabase client в новой версии?»

### Tavily / Exa
Используй, когда нужен поиск по вебу, сравнение подходов, статьи, гайды, обсуждения.

Пример:
- «Какие есть способы хранить long-term memory для AI agent?»
- «Сравни 3 популярных подхода к Telegram Mini Apps auth»

### Local / semantic code search
Используй, когда ответ уже может быть внутри проекта.

Пример:
- «Где в этом репозитории реализована авторизация?»
- «Найди, как здесь уже работают с retry логикой»

### Playwright
Используй, когда нужно не просто прочитать код, а проверить интерфейс или поведение страницы.

Пример:
- «Открой localhost:3000 и проверь ошибки в консоли»
- «Посмотри, почему кнопка не нажимается»

---

## ⚖️ Главное правило выбора
Не используй сложный инструмент, если хватает более простого.

Например:
- если нужен ответ по библиотеке → сначала Context7, а не общий веб-поиск
- если нужно понять локальную реализацию → сначала поиск по коду, а не интернет
- если надо проверить UI → Playwright, а не догадки по JSX

---

## ✅ Проверка результата
После поиска перепроверь ключевой вывод минимум по одному надёжному источнику.

Особенно если:
- вопрос связан с безопасностью
- API мог поменяться
- ответ влияет на архитектурное решение

---

## ✅ Чек-лист для самопроверки
- [ ] Я понимаю разницу между docs-поиском, веб-поиском, локальным поиском и браузерной проверкой
- [ ] Я не использую тяжёлый инструмент без необходимости
- [ ] Я сначала ищу в проекте, если ответ может уже быть внутри кода
- [ ] Я перепроверяю важные выводы по надёжному источнику
