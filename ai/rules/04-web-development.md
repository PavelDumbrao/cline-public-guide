# 🌐 Веб-разработка

> Это правило нужно, чтобы агент не делал «просто красивый экран», а создавал интерфейс, который реально понятен пользователю и не ломает проект.

- [`06-code-quality.md`](06-code-quality.md) — как не испортить качество кода ради быстрой вёрстки

---

## Зачем это правило нужно

Новички часто попадают в одну из двух крайностей:
- либо делают интерфейс «как-нибудь, лишь бы работало»
- либо начинают украшать и усложнять без пользы

Это правило задаёт здравый баланс:
- интерфейс должен быть понятным
- реализация должна соответствовать стеку проекта
- агент не должен придумывать несуществующий backend или API

---

## 🎨 Интерфейс
- Дефолтно делай UI понятным и нейтральным.
- Учитывай адаптивность и доступность.
- Не добавляй маркетинговый мусор без запроса.

### Что это значит на практике
- кнопки должны быть понятны по тексту
- форма должна быть читаема и на телефоне, и на десктопе
- не вставляй лишние рекламные блоки, баннеры и продающие тексты, если об этом не просили

---

## 🧱 Перед реализацией
- Проверь стек проекта.
- Уточни must-have сценарии пользователя.
- Не придумывай API, если его нет.

### Мини-примеры
Правильно:
- сначала понять, проект на React, Next.js, Vue или обычном HTML
- сначала выяснить, что пользователь хочет: лендинг, форму, дашборд, каталог

Неправильно:
- начать писать компонент под Tailwind, если в проекте его нет
- придумать endpoint `/api/submit`, если backend нигде не описан

---

## ✅ Проверка
- Проверяй результат в браузере или через тесты.
- Следи за консолью и сетевыми ошибками.
- Не считай задачу завершённой только потому, что код «выглядит нормально».

---

## ✅ Чек-лист для самопроверки
- [ ] Я понимаю, какой стек используется в проекте
- [ ] Интерфейс получился понятным, а не просто «красивым»
- [ ] Я не придумал несуществующий API
- [ ] После изменений я реально проверил результат в браузере или тестами
