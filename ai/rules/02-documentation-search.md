# 📚 Поиск документации

> Это правило нужно, чтобы агент не фантазировал там, где ответ можно и нужно проверить по документации.

Сначала полезно прочитать:
- [`05-search-tools.md`](05-search-tools.md) — какой инструмент поиска когда использовать
- [`../../docs/07-mcp-guide-human.md`](../../docs/07-mcp-guide-human.md) — как понимать MCP и search-инструменты как ученик

---

## Зачем это правило нужно

Одна из самых частых ошибок — уверенно писать код по памяти там, где библиотека уже изменилась.

Это правило напоминает:
- если есть сомнение — проверь
- если API мог поменяться — проверь
- если синтаксис важен — проверь

---

## Когда искать
- Если не знаешь точный API.
- Если нужна актуальная синтаксическая деталь.
- Если библиотека быстро меняется.
- Если пользователь просит сделать что-то в незнакомом инструменте.

### Мини-примеры
Искать обязательно, если:
- нужно настроить новую библиотеку
- надо вспомнить точный синтаксис метода
- речь идёт о версии фреймворка, которая могла измениться

---

## 🧭 Что искать через docs, а что — через веб

Через **документацию** ищи:
- API библиотек
- официальные параметры конфигурации
- примеры из официальных гайдов

Через **веб-поиск** ищи:
- сравнение подходов
- обсуждения и реальные кейсы
- обходные пути и практические статьи

Простое правило:
- нужен точный синтаксис → docs
- нужен обзор и варианты → web search

---

## Приоритеты
1. Официальная документация
2. Официальные руководства
3. Авторитетные технические статьи
4. Проверенные примеры кода

---

## ⚠️ Главное правило
Не выдумывай конфигурацию библиотек, если её можно проверить по docs.

Неправильно:
- «Наверное, тут параметр называется `useCache: true`»

Правильно:
- «Сначала проверю документацию, как именно называется этот параметр в текущей версии библиотеки.»
