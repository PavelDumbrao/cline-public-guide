# Systematic Debugging

> Этот skill нужен, когда проблему нельзя решить «на глаз» и важно найти **реальную причину**, а не просто замаскировать симптом.

Сначала полезно прочитать:
- [`../../rules/06-code-quality.md`](../../rules/06-code-quality.md) — чтобы не чинить баг ценой ухудшения качества кода
- [`../smoke-test/SKILL.md`](../smoke-test/SKILL.md) — чтобы после фикса проверить, что всё правда работает

---

## Что это за skill простыми словами

Этот skill не про хаотичное «давай попробуем ещё вот это».
Он про последовательный дебаг:
- воспроизвести проблему
- собрать факты
- сформировать гипотезы
- проверить их по одной
- исправить именно причину

Если ученик научится использовать этот skill правильно, он перестанет лечить баги вслепую.

---

## Когда вызывать
- баг воспроизводится, но причина неочевидна
- система ведёт себя нестабильно
- нужно найти root cause, а не замаскировать симптом
- после неудачных хаотичных попыток фикса

## Когда НЕ вызывать
- задача не про баг, а про новую фичу
- проблема уже точно локализована и требуется только механическая правка
- нет минимального способа воспроизведения и сначала нужно собрать факты

## Decision Tree

### 1. Баг воспроизводится?
- Да → иди в изоляцию
- Нет → сначала собери данные, логи, условия, окружение

### 2. Есть stack trace / error log?
- Да → начни с первой реальной ошибки
- Нет → сначала добавь наблюдаемость

### 3. Проблема в данных, логике или окружении?
- Данные → проверь входы, null/undefined, формат, schema
- Логика → проверь условия, ветвления, state
- Окружение → проверь env, версии, сеть, доступы

### 4. После фикса баг исчез?
- Да → проверь регрессию
- Нет → вернись к гипотезам, не меняя всё подряд

## Порядок диагностики
1. Воспроизведи проблему
2. Зафиксируй ожидаемое vs фактическое поведение
3. Собери логи, stack trace, контекст
4. Изолируй участок
5. Сформируй 2–3 гипотезы
6. Проверяй гипотезы по одной
7. Найди причину
8. Исправь причину
9. Проверь регрессию
10. Кратко оформи вывод

## Как формулировать гипотезы
Хорошая гипотеза:
- конкретная
- проверяемая
- привязана к наблюдаемому симптому

### Примеры хороших гипотез
- `bio` приходит как `undefined`, а код вызывает `.length`
- запрос падает из-за отсутствующего header авторизации
- ошибка вызвана race condition между двумя эффектами

### Плохие гипотезы
- «что-то не так с сервером»
- «возможно баг в библиотеке» без проверки
- «давай попробуем переписать полфайла»

## Как остановить бессмысленное тыкание во всё подряд
- меняй только одну вещь за раз
- после каждой проверки фиксируй результат
- не прыгай к новому подозрению без вывода по предыдущему
- если сделал 3 хаотичные правки подряд — остановись и вернись к фактам

## Проверка регрессии
После фикса обязательно проверь:
- исходный баг исчез
- соседний сценарий не сломался
- негативный сценарий обрабатывается адекватно
- если возможно, есть тест или smoke check

## Как оформлять итог дебага
Не пиши размытое «починил».
Оформи результат так, чтобы другой человек понял ход мысли.

### Output format
- **Проблема:** что именно ломалось
- **Гипотезы:** 2–3 проверяемые версии
- **Проверка:** что проверялось и что подтвердилось/не подтвердилось
- **Причина:** реальный root cause
- **Исправление:** что изменено
- **Как проверено:** чем подтверждено, что фикс работает

## Типичные ошибки
- менять много вещей сразу
- игнорировать первую ошибку в trace
- считать, что "похоже работает" = исправлено
- путать симптом и причину
- не проверять регрессию

## Как проверить результат skill
- повторяется ли исходный сценарий без ошибки
- есть ли ясный root cause
- есть ли проверка после фикса
- исчез ли хаотичный стиль дебага
