## 🧭 Как эта система реально собирается по слоям

Самая частая ошибка ученика — пытаться понять всё сразу в случайном порядке.
Нормальная последовательность такая:

### Слой 1 — Rules
Сначала настраиваешь **always-on поведение**:
- язык,
- стиль,
- поиск документации,
- использование инструментов,
- базовую безопасность,
- управление skills.

Именно Rules уменьшают хаос в первых сессиях.

### Слой 2 — Skills
Потом добавляешь **on-demand экспертизу**:
- systematic-debugging,
- frontend-design,
- deployment-guide,
- local-code-search,
- n8n-* и другие domain-specific skills.

Skills не должны тащиться в контекст всегда. Они хороши именно тем, что подключаются по ситуации.

### Слой 3 — MCP
Потом подключаешь внешние инструменты:
- Context7,
- Tavily,
- Playwright,
- GitHub и другие.

MCP расширяют возможности агента, но сами по себе не заменяют ни Rules, ни Skills.

### Слой 4 — Hooks
И только после этого добавляешь hooks.

Почему hooks последними:
- они автоматические;
- они могут ломать workflow, если непонятны;
- их нужно ставить тогда, когда ты уже понимаешь, зачем именно тебе нужна автоматизация событий.

### Слой 5 — Workflows
Когда базовые слои уже понятны, добавляешь reusable workflows.

Почему workflows не в самом начале:
- workflow имеет смысл только тогда, когда ты уже понимаешь, какие шаги у тебя повторяются;
- раньше этого ты рискуешь зафиксировать сырой или случайный процесс;
- зрелый workflow должен опираться на уже понятные rules, skills, MCP и при необходимости hooks.

### ✅ Чек-лист для самопроверки

- [ ] Я не начинаю с hooks раньше, чем понял базовый setup
- [ ] Я могу объяснить, что именно даёт каждый слой системы

## 🧩 Главное, что нужно понять про сущности

Часто путают — давай разберёмся раз и навсегда:

| Сущность | Что это на самом деле | Где лежит |
|---|---|---|
| **Rule** | Постоянное правило поведения агента. Работает всегда. | `ai/rules/` |
| **Skill** | Специализированная инструкция для конкретной задачи. Активируется по запросу. | `ai/skills/` |
| **Hook** | Bash-скрипт, который срабатывает на событиях Cline (старт задачи, завершение и т.д.) | `ai/hooks/` |
| **Cline Workflow** | Markdown-процесс — инструкция для агента по сложной многошаговой задаче | `ai/workflows/` |
| **n8n Workflow** | JSON-файл для импорта в n8n. К Cline прямого отношения не имеет. | `examples/n8n/` |

> Cline Workflows и n8n Workflows — **это принципиально разные вещи**. Не путай.

## 🧠 Большая карта: что из этого работает всегда, а что по запросу

| Слой | Работает всегда? | Для кого | Что решает |
|---|---|---|---|
| **Rules** | Да | Агент | Базовая политика, поведение, routing |
| **Skills** | Нет, по ситуации | Агент | Глубокая экспертиза по отдельной теме |
| **Hooks** | Автоматически по событию | Система вокруг агента | Блокировки, логирование, recovery, automation |
| **MCP** | Только когда вызываются | Агент | Доступ к внешним данным и инструментам |
| **Workflows** | Только когда запускаются | Агент / человек | Повторяемые многошаговые процедуры |

Если коротко:
- **Rules** — характер и политика;
- **Skills** — экспертиза;
- **Hooks** — автоматика;
- **MCP** — внешние руки и глаза;
- **Workflows** — повторяемые процессы.

Если путаешься, открой:
- [`docs/23-system-map-rules-skills-hooks-mcp.md`](docs/23-system-map-rules-skills-hooks-mcp.md)

## 🔒 Что сознательно не включено

Это публичный репозиторий — шаблон, а не копия живой системы.
Поэтому здесь нет:

- личного `USER.md` (у тебя он будет свой)
- личного `tech-stack.md` (твой стек — твои правила)
- реальных токенов, IP, доменов
- приватных заметок об инцидентах
- живых credential-блоков из workflow JSON

Также здесь сознательно не включены:
- твои личные habits и routing overrides, если они не reusable;
- случайные локальные ошибки, которые ещё не были проверены и не стали устойчивыми уроками;
- hooks/skills/rules, которые заточены под одну приватную инфраструктуру без обезличивания.

**Всё это ты добавишь сам, под свою систему.**

## 💡 Почему именно такая система

Этот репозиторий — результат месяцев реальной работы с Cline в клубе ProAiCommunity.
Мы проходили через хаос, нестабильные результаты, потерянный контекст и агентов, которые делали не то.
Вся эта система — дистилляция того, что реально работает.

Важно: она развивается не хаотично, а по принципу:
- ошибка → исправление → проверка → урок → правильный слой фиксации.

То есть не каждая проблема рождает новый skill или новый rule.
Сначала нужно понять, где это знание должно жить:
- в Rule,
- в Skill,
- в Hook,
- в Workflow,
- или просто в проектной заметке.

Теперь она твоя. Бери, адаптируй, используй.
