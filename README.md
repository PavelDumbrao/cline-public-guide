Ты только начинаешь работать с Cline — AI-помощником внутри VS Code.
Или уже работаешь, но хаотично: результаты непредсказуемые, агент делает что-то не то, контекст теряется.

**Этот репозиторий решает именно эту проблему.**

Здесь собрана система настройки Cline — та самая, которую мы используем в клубе ProAiCommunity для реальных проектов: автоматизации, AI-агентов, n8n, Telegram-ботов и всего остального.

Всё обезличено и готово к использованию. Никаких личных токенов, IP и секретов — только структура, правила и примеры, которые ты адаптируешь под себя.

## 🧭 С чего начать — зависит от тебя

### Хочу, чтобы мой бро Cline сам помог мне всё собрать
Этот репозиторий сделан не только для тебя как для человека.
Он сделан для связки:
- **ты** — человек;
- **твой бро Cline** — ИИ-агент, который помогает тебе всё настроить.

То есть твоя задача простая:
не сидеть и не разбираться во всём в одиночку,
а дать этот репозиторий своему Cline и отправить ему не просто сообщение, а сильный мега-промпт.

**Что делает человек:**
1. Открывает чат с Cline.
2. Даёт ему ссылку на этот репозиторий.
3. Копирует мега-промпт ниже.
4. Отправляет его Cline.
5. Дальше идёт с ним шаг за шагом.

**Ссылка на репозиторий:**
- `https://github.com/PavelDumbrao/cline-public-guide`

### Что важно подготовить до запуска мега-промпта
- У тебя должен открываться сам Cline.
- Желательно, чтобы GitHub уже был доступен в Cline.
- Если агент просит GitHub MCP, Exa, Tavily или другой MCP — сначала поставь нужный сервер.
- Если нужен токен — не ищи его сам вслепую, а проси агента дать тебе **прямую ссылку**, где его взять.
- После установки нового MCP проверяй, появился ли он в интерфейсе. Если не появился — агент должен сказать, нужен ли reload или reopen окна.

### Как человеку вести себя во время настройки
- Не смотри только в чат — смотри ещё в **Rules**, **Skills**, **MCP**, **Configured**.
- Если агент пишет «готово», а ты этого не видишь в интерфейсе — значит, ещё не готово.
- Не давай агенту одновременно настраивать один и тот же setup в двух окнах.
- Не пугайся промежуточных ошибок: важен не сам факт ошибки, а пришёл ли агент к рабочему результату.
- Если что-то не видно, говори конкретно: **не вижу rule**, **не вижу skill**, **не вижу MCP в Configured**.

### Главное правило старта
- Этот repo — **не готовая установка**, а инструкция и база для настройки.
- Один вставленный мега-промпт не означает, что всё само уже правильно поставилось.
- После каждого шага человек должен глазами проверить интерфейс и подтвердить результат.
- Пока ты не увидел rule / skill / MCP в интерфейсе — считай, что шаг ещё не завершён.

## 🤖 Мега-промпт для Cline ИИ Агента ученика
**Скопируй этот мега-промпт целиком и вставь в чат своему Cline.**

Это не объяснение для человека.
Это не просто пример промпта. Это сильный мега-промпт для твоего ИИ-агента, который должен помочь тебе спокойно и по шагам собрать рабочую систему. Если ученик закинет в Cline слабую короткую формулировку, агент легко сделает всё криво, кусками или без нормальной проверки.

```text
Изучи этот репозиторий как учебный шаблон по настройке Cline:
https://github.com/PavelDumbrao/cline-public-guide

Эта инструкция написана для меня, человека, и для тебя, моего ИИ-помощника Cline.
Твоя задача — не просто посмотреть этот repo, а помочь мне собрать мой рабочий setup спокойно, понятно и по шагам.

Работай простым и понятным языком.
Не пытайся сделать всё сразу.
Не копируй чужую систему один в один.
Не ставь лишнее.
Не говори «готово», пока я не увижу результат в интерфейсе.

Сначала проверь базовую готовность:
1. Убедись, что у тебя есть доступ к этому repo.
2. Проверь, хватает ли тебе MCP для чтения репозитория, документации и поиска.
3. Если тебе нужен токен или доступ, не просто попроси его, а сначала дай мне прямую ссылку, где это взять.
4. Если после установки MCP нужен reload или reopen окна — скажи об этом прямо.

Сначала сделай вот что:
1. Изучи главные файлы репозитория:
   - README.md
   - docs/24-rules-mega-guide.md
   - docs/25-cross-platform-installation-paths.md
   - docs/26-mega-prompt-for-ai-installer.md
   - docs/27-setup-self-check-and-recovery.md
   - docs/23-system-map-rules-skills-hooks-mcp.md
2. Потом объясни мне простыми словами:
   - какие правила мне реально нужны для старта;
   - какие скиллы мне реально нужны;
   - нужны ли мне хуки вообще;
   - какие MCP нужны сразу, а какие пока не нужны.

Дальше работай так:
1. Сначала сделай короткий аудит.
2. Потом дай мне простой план.
3. Потом создай чек-лист всей задачи.
4. Потом предложи мне минимальный стартовый набор.
5. Только после этого помогай ставить всё по шагам.

Порядок установки:
1. rules
2. skills
3. MCP
4. hooks
5. workflows

Очень важно по rules:
- установи все базовые rules по списку из гайда, а не 2–3 тестовых rules;
- после установки перечисли мне, какие именно rules уже установлены;
- попроси меня проверить их в интерфейсе;
- если я не вижу часть rules, не переходи к следующему шагу.

Очень важно по skills:
- если ты пишешь, что skill создан, а я не вижу его в Skills, считай, что задача не выполнена;
- не спорь со мной, а перепроверь путь, формат и место создания;
- если нужно, создай тестовый skill как проверку и найди реальный рабочий путь;
- только после подтверждения через интерфейс переходи дальше.

Очень важно по MCP:
- подключай MCP по одному;
- после каждого MCP проверяй, появился ли он в Configured или в рабочем списке серверов;
- если MCP поставлен, но не виден, скажи, нужен ли reload / reopen окна;
- не иди дальше, пока текущий MCP реально не подтверждён.

После каждого шага ты обязан:
- сказать простыми словами, что сделал;
- показать, какие файлы изменил;
- перечитать их обратно;
- проверить, что это реально заработало;
- не идти дальше, пока прошлый шаг не подтверждён.

Если тебе нужны мои данные, например:
- токен;
- логин;
- API key;
- ссылка;
- доступ;
то не выдумывай их, а остановись, дай прямую ссылку, где это взять, и скажи, что мне нужно сделать вручную.

Если не уверен, куда именно ставить файлы, тоже остановись и сначала разберись.
Если я не вижу результат в интерфейсе, считай, что шаг не завершён.
Работай с этой настройкой в одном активном окне, а не в нескольких сразу.

Сразу в начале работы:
1. сделай аудит;
2. дай план;
3. создай чек-лист;
4. предложи минимальный setup.

Потом помогай мне идти шаг за шагом.
```

## 🧭 Как эта система реально собирается по слоям

Самая частая ошибка ученика — пытаться понять всё сразу в случайном порядке.
Нормальная последовательность такая:

### Слой 1 — Rules
Сначала настраиваешь **always-on поведение**:
- язык,
- стиль,
- поиск документации,
- использование инструментов,
- базовую безопасность,
- управление skills.

Именно Rules уменьшают хаос в первых сессиях.

### Слой 2 — Skills
Потом добавляешь **on-demand экспертизу**:
- systematic-debugging,
- frontend-design,
- deployment-guide,
- local-code-search,
- n8n-* и другие domain-specific skills.

Skills не должны тащиться в контекст всегда. Они хороши именно тем, что подключаются по ситуации.

### Слой 3 — MCP
Потом подключаешь внешние инструменты:
- Context7,
- Tavily,
- Playwright,
- GitHub и другие.

MCP расширяют возможности агента, но сами по себе не заменяют ни Rules, ни Skills.

### Слой 4 — Hooks
И только после этого добавляешь hooks.

Почему hooks последними:
- они автоматические;
- они могут ломать workflow, если непонятны;
- их нужно ставить тогда, когда ты уже понимаешь, зачем именно тебе нужна автоматизация событий.

### Слой 5 — Workflows
Когда базовые слои уже понятны, добавляешь reusable workflows.

Почему workflows не в самом начале:
- workflow имеет смысл только тогда, когда ты уже понимаешь, какие шаги у тебя повторяются;
- раньше этого ты рискуешь зафиксировать сырой или случайный процесс;
- зрелый workflow должен опираться на уже понятные rules, skills, MCP и при необходимости hooks.

### ✅ Чек-лист для самопроверки
- [ ] Я понимаю, что Rules, Skills, MCP и Hooks добавляются не одновременно
- [ ] Я не начинаю с hooks раньше, чем понял базовый setup
- [ ] Я могу объяснить, что именно даёт каждый слой системы

## 🧩 Главное, что нужно понять про сущности

Часто путают — давай разберёмся раз и навсегда:

| Сущность | Что это на самом деле | Где лежит |
|---|---|---|
| **Rule** | Постоянное правило поведения агента. Работает всегда. | `ai/rules/` |
| **Skill** | Специализированная инструкция для конкретной задачи. Активируется по запросу. | `ai/skills/` |
| **Hook** | Bash-скрипт, который срабатывает на событиях Cline (старт задачи, завершение и т.д.) | `ai/hooks/` |
| **Cline Workflow** | Markdown-процесс — инструкция для агента по сложной многошаговой задаче | `ai/workflows/` |
| **n8n Workflow** | JSON-файл для импорта в n8n. К Cline прямого отношения не имеет. | `examples/n8n/` |

## 🧠 Большая карта: что из этого работает всегда, а что по запросу

| Слой | Работает всегда? | Для кого | Что решает |
|---|---|---|---|
| **Rules** | Да | Агент | Базовая политика, поведение, routing |
| **Skills** | Нет, по ситуации | Агент | Глубокая экспертиза по отдельной теме |
| **Hooks** | Автоматически по событию | Система вокруг агента | Блокировки, логирование, recovery, automation |
| **MCP** | Только когда вызываются | Агент | Доступ к внешним данным и инструментам |
| **Workflows** | Только когда запускаются | Агент / человек | Повторяемые многошаговые процедуры |

Если коротко:
- **Rules** — характер и политика;
- **Skills** — экспертиза;
- **Hooks** — автоматика;
- **MCP** — внешние руки и глаза;
- **Workflows** — повторяемые процессы.

Если путаешься, открой:
- [`docs/23-system-map-rules-skills-hooks-mcp.md`](docs/23-system-map-rules-skills-hooks-mcp.md)

## 🔒 Что сознательно не включено

Это публичный репозиторий — шаблон, а не копия живой системы.
Поэтому здесь нет:

- личного `USER.md` (у тебя он будет свой)
- личного `tech-stack.md` (твой стек — твои правила)
- реальных токенов, IP, доменов
- приватных заметок об инцидентах
- живых credential-блоков из workflow JSON

Также здесь сознательно не включены:
- твои личные habits и routing overrides, если они не reusable;
- случайные локальные ошибки, которые ещё не были проверены и не стали устойчивыми уроками;
- hooks/skills/rules, которые заточены под одну приватную инфраструктуру без обезличивания.

**Всё это ты добавишь сам, под свою систему.**

## 💡 Почему именно такая система

Этот репозиторий — результат месяцев реальной работы с Cline в клубе ProAiCommunity.
Мы проходили через хаос, нестабильные результаты, потерянный контекст и агентов, которые делали не то.
Вся эта система — дистилляция того, что реально работает.

Важно: она развивается не хаотично, а по принципу:
- ошибка → исправление → проверка → урок → правильный слой фиксации.

То есть не каждая проблема рождает новый skill или новый rule.
Сначала нужно понять, где это знание должно жить:
- в Rule,
- в Skill,
- в Hook,
- в Workflow,
- или просто в проектной заметке.

Теперь она твоя. Бери, адаптируй, используй.
